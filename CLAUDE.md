# アプリケーションの概要
音声処理アプリケーション。
ローカル環境で動作するアプリとして開発する。詳細仕様(処理タイプ、入出力フォーマット、リアルタイム性等)はこの後詰める。

# 前提
- **コミュニケーションは日本語**で行う。
- **アーキテクチャを意識**する。役割ベースの分割を優先する。
- **クラスの役割を決めて表明**する。表明先は以下の2か所:
  - 該当クラスのコメント/Docstring の冒頭に1〜2行で役割を明記。
  - `docs/design/Class.md` のクラス一覧に役割を記載(レイヤ単位の責務は `Architecture.html`)。
- 役割に沿って拡張し、異なる役割とみなせる場合は**役割を分割**する。
- クラスは可能な限り**単一責任原則**を順守する。
- 何かを変更するとき最短の手法よりも長期的に使える手法を考える。

# 配布方針 (公開を見据えた前提)
- **将来 GitHub で公開**して誰でも持っていける形を目指す。特定ハード必須は避ける。
- **CPU を floor** とする: NVIDIA GPU 無しでも起動・縦通しが可能であること。
- **GPU/アクセラレータがあれば自動で使う**(NVIDIA CUDA / Apple Silicon MPS 等)。**device 切替はコードパスを1本に保つ**(条件分岐ではなく `device="auto"` の引数で吸収)。
- 「GPU 専用バックエンド」と「CPU 専用バックエンド」を別実装として並列で持たない(テストが2倍になる)。
- 詳細・再検討トリガは `docs/design/pendList.md` の方針エントリ「プロジェクト配布方針」参照。

# 使用技術
- **言語**: Python 3.x
- **依存/プロジェクト管理**: uv (Rust実装の高速ツール。pip + venv + poetry + pyenv 相当を1つに統合)
  - `pyproject.toml` ベース、ロックは `uv.lock`
  - インタプリタ管理も uv に任せる(セットアップは `uv sync` 一発)
- **GUI**: customtkinter (Tkinter ベースのモダン版、追加配布物が軽い)
- **音声取得**: soundcard (クロスOS、loopback対応。MVPはWindows優先)
- **VAD**: Silero-VAD (発話区切り検出)
- **ASR**: faster-whisper (Whisper高速実装。タスクは `transcribe` 固定。translate不使用)
- **翻訳**: NLLB-200 (distilled 600M, Meta製, MIT, 200言語対応, ローカル動作)
- **TTS**: Windows SAPI (pyttsx3 経由) を仮実装。後で VOICEVOX/ゆっくり系等に差し替え可能とする
- **音声フォーマット内部標準**: 16kHz / mono / float32(リサンプリングは入力レイヤの責務)

各レイヤは Strategy/Adapter パターンで実装を差し替え可能にする(MVPは1実装のみ定義)。

```
docs/
├── manual.md                  … アプリケーションの使い方
└── design/
    ├── Architecture.html      … アーキテクチャ(レイヤ図・役割・I/F)
    ├── Class.md               … クラス/モジュール詳細(Architectureから参照)
    ├── UserSinario.md         … ユーザシナリオ
    ├── TaskList.md            … 実装タスクリスト(全体)
    ├── pendList.md            … 保留・暫定決定リスト
    ├── <branch-name>/         … 進行中の作業ブランチごとのフォルダ
    │   ├── Plan.md            … 当該ブランチで行う作業計画(or .html)
    │   └── testPlan.md        … 当該ブランチのテスト項目
```

「進行中」と「完了」を視覚的に分離するため)。トップレベルの md/html は常に最新版を維持。

※ 本アプリは DB を持たないため `DataModel.md` は作成しない(必要になったら追加)。
※ `Architecture` は図を含むため html、`Class` はテキスト中心のため md。

## pendList.md の用途
以下のような「**その時点では対応しないが忘れたくない項目**」を記録する:
- 実装後に発覚した規模の大きな変更
- ドッグフーディングで見つかった挙動の修正
- 設計上の暫定決定で、後日見直したいもの

各項目は `[起票日] 内容 / 背景 / 対応の見送り理由` を含める。

# テスト方針
- **Googleのテストエンジニアリング**に従う。
- **テストピラミッド**を意識する。
- **small テストで多くをテスト**する。
- **middle / large は基本的に正常系**を扱う。
- **network/DB/システム間I/F は異常系も十分に想定**する。
- 本アプリはローカル動作のため、**テストもローカル実行で完結**させる(リモートCI不要)。

## テストツール
- **ランナー**: pytest
- **プラグイン**: `pytest-cov`(カバレッジ計測) / `pytest-mock`(モック注入)
- **モック方針**: 各レイヤのバックエンドはI/Fで分離されているため、単体テストは具象を差し替えて行う(`pytest-mock`)。
- **音声I/O テスト**: 実機依存を避けるため、**録音済みWAVを入力にしたE2Eテスト**を主軸にし、各バックエンドはsmallテストで個別検証する。
- **カバレッジ目標**: [TBD: 実装が進んだ時点で数値設定]

## テスト階層 (small / middle / large)
Google テストピラミッドに対応した pytest マーカーで階層を表現する。

| 階層 | マーカー | 内容 | 実行タイミング |
|---|---|---|---|
| **small** | (なし) | モック中心、I/O なし、< 1 秒/件 | 毎回(default) |
| **middle** | `@pytest.mark.middle` | 実コンポーネントだが軽量(ConfigStore/Logger 実体、WAV ファイル読込 等) | リリース前 / 機能更新時 |
| **large** | `@pytest.mark.large` | 実モデル/実デバイス必須(NLLB-200 推論、Whisper 推論、実マイク等) | 手動で明示実行のみ |

### 実行コマンド
```bash
py -m uv run pytest                                 # small のみ(既定で middle/large 除外)
py -m uv run pytest -m middle                       # middle のみ
py -m uv run pytest -m large                        # large のみ
py -m uv run pytest -m "middle or large"            # リリース前の確認用
py -m uv run pytest -m "" --override-ini="addopts=" # 全部(addopts を強制 override)
```

### マーカー付与の方針
- 新規テストは原則 small。実装が要求する場合のみ middle/large を選ぶ。
- middle: 「実コンポーネントだが MVP 環境内で完結し、数秒以内に終わる」
- large: 「外部リソース(モデル DL 済み前提 / 実デバイス)を必要とし、数十秒以上かかる」
- large は **CI に組み込まない**(リモート CI 不要方針)。手元での明示確認のみ。

# 実装について
- `/src` 以下にコード・プロジェクトを配置する。
- **基本は設計に沿う**。役割分担に従って実装する。
- 実装後に設計と異なる箇所がある場合、**ズレの種類に応じて対処を分ける**:
  - **役割の解釈ミス・細部の逸脱** → 実装をリファクタリングして設計に合わせる。
  - **フレームワーク制約・性能要件 等で設計が現実的でない** → 設計ドキュメント側を更新する。
- 役割と使用フレームワークに沿った**テスト項目を抽出**する。

## 運用

### 作業
- ユーザが依頼するまでは**クラスやメソッドは必要最小限**に抑え、**機能や役割を中心に会話**する。

### 長文応答の取り扱い
- チャット上の応答が **20行を超える** 場合は、本文を **ワークフォルダ直下の `tmp/reportN.md`** に書き出し、チャット上は要点と参照リンクだけに留める。
  - 保存先ディレクトリ `tmp/` が無ければ作成する。**システムの `/tmp` は使わない**(プロジェクト直下に閉じる)。
  - 出力形式は **Markdown(`.md`)** を採用する。理由: プロジェクト全体がmd統一/エディタで素のままプレビューでき、編集・diff・grepしやすい/表やコードブロックも十分扱える。
  - ファイルは毎回上書きしてよい(履歴が必要な場合はユーザが明示する)。
  - チャット側には少なくとも「保存先パス」「中身の見出し一覧 or 1〜3行サマリ」を残す。
  - レビュー結果・設計案・調査結果など、長くなりがちな出力で適用する。
- `tmp/` は作業用一時領域。**git 管理対象外**(`.gitignore` に追加)。

### コミット・ブランチ
- **編集後の自動コミット**: ファイルを編集してユーザに制御を返すタイミングで、更新があれば必ず git にコミットする。タイミングは厳密でなくてよい(複数の作業をまとめて1コミットでもよい)が、未コミットのまま長時間放置しない。
- **コミット前にビルドとテストを通す**(コード作成段階以降)。
- **ブランチ運用**: 機能実装の前に必ずブランチを切る。`master` / `main` には原則レビュー済みのものだけが入る前提で運用する。
  - **ブランチプレフィックス**: 下記の「プレフィックス対応表」を参照。ブランチ名は `feature/phase02-todo-mvp` のように内容が分かるもの。
  - **ドキュメント編集だけの場合**はブランチを切らず、直接 `master` にコミットしてよい(コミットプレフィックスは `docs:`)。
  - **サイクル**: ブランチ作成 → 作業 → ユーザレビュー → マージ。レビューは即時に行えないことがあり、後でまとめて行うこともある。
- **マージ運用**:
  - **マージはユーザの依頼があってから**実施する。例外として「Phase X までまとめて進めて」のような明示的な指示があれば、その範囲で各 Phase 完了時にマージしてよい。
  - **マージは必ず `--no-ff` を使う**(マージコミットを残してブランチ単位の作業履歴を保つ)。
- **リモート操作**:
  - **リモートブランチには絶対に触らない**(`git push`、`push --force`、リモートを参照する `rebase` 等は行わない)。
  - ローカルブランチ・タグは自由に使ってよい。

#### プレフィックス対応表
ブランチとコミットでプレフィックスを揃える(`/` と `:` の違いのみ)。

| 種類             | ブランチ      | コミット   |
| ---------------- | ------------- | ---------- |
| 新機能           | `feature/`    | `feat:`    |
| 不具合修正       | `bugfix/`     | `fix:`     |
| リファクタリング | `refactor/`   | `refactor:`|
| ドキュメント変更 | `docs/`       | `docs:`    |
| テスト追加・修正 | `test/`       | `test:`    |
| 雑務             | `chore/`      | `chore:`   |

#### コミットメッセージ形式
```
prefix: 大まかな概要(100文字以内)

詳細: ふるまい・抽象レベルでの変更内容と「なぜ」を中心に。
```

**書かない**:
- 個別のファイル名・クラス名・メソッド名(コードを見ればわかる)
- テスト結果(基本 100% pass が前提。特殊な事情が無ければ書かない)
- 行数・差分サマリ(`git show --stat` で見れば足りる)

**書く**:
- 「何が出来るようになったか / 何が変わったか」を抽象レベルで
- そう決めた理由(複数案の比較、トレードオフ、配布方針との関係 等)
- 副作用や注意点(マイグレーション・破壊的変更・後で消すべき暫定処置 等)
