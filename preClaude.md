## 運用

### 作業
- ユーザが依頼するまではクラスやメソッドは必要最小限に抑え、機能や役割を中心に会話する。

### コミット・ブランチ
- **編集後の自動コミット**: ファイルを編集してユーザに制御を返すタイミングで、更新があれば必ず git にコミットする。タイミングは厳密でなくてよい(複数の作業をまとめて1コミットでもよい)が、未コミットのまま長時間放置しない。
- **コミット前にビルドとテストを通す**(コード作成段階以降)。
- **ブランチ運用**: 機能実装の前に必ずブランチを切る。`master` / `main` には原則レビュー済みのものだけが入る前提で運用する。
  - **ブランチプレフィックス(例)**: `feature/`(新機能)、`refactor/`(リファクタ)、`bugfix/`(不具合修正)、`docs/`(ドキュメント変更)、`chore/`(雑務)。ブランチ名は `feature/phase02-todo-mvp` のように内容が分かるもの。
  - **ドキュメント編集だけの場合**はブランチを切らず、直接 `master` にコミットしてよい(コミットプレフィックスは `docs:`)。
  - **サイクル**: ブランチ作成 → 作業 → ユーザレビュー → マージ。レビューは即時に行えないことがあり、後でまとめて行うこともある。
- **マージ運用**:
  - **マージはユーザの依頼があってから**実施する。例外として「Phase X までまとめて進めて」のような明示的な指示があれば、その範囲で各 Phase 完了時にマージしてよい。
  - **マージは必ず `--no-ff` を使う**(マージコミットを残してブランチ単位の作業履歴を保つ)。
- **リモート操作**:
  - **リモートブランチには絶対に触らない**(`git push`、`push --force`、リモートを参照する `rebase` 等は行わない)。
  - ローカルブランチ・タグは自由に使ってよい。
