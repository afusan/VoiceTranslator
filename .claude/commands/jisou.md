リクエスト: $ARGUMENTS

あなたは主エージェント(オーケストレータ)として、上のリクエストを **自走** で実装まで回せ。
運用は `docs/design/AgenticWorkflow.html` の規律に従う。**原則「クリティカル時以外は止まらない」**。

## 手順

進捗を `TaskCreate` で起こし、各フェーズを `in_progress` / `completed` で更新しながら進める。

### 0a. ブランチと作業ディレクトリ
- 現在のブランチを確認:
  - `master` / `main` にいる場合のみ、内容を表す名前で新ブランチを切る
    (`feature/` / `bugfix/` / `refactor/` / `docs/` / `test/` / `chore/` のいずれか)。
  - 既に作業ブランチにいるなら、その上で続行。
- 対象 design ディレクトリ `docs/design/<branch-name>/` を作成。

### 0b. request.md の保存
- 起動時の **入力プロンプト全文**(上の「リクエスト:」以降)を、そのまま
  `docs/design/<branch>/request.md` として保存する。
- 改変・要約はしない(後で利用者が見返せるよう原文を残す)。
- **コミット**: `git add docs/design/<branch>/request.md && git commit -m "docs: request 記録"`。

### 0c. 要求の確認と checkedRequest.md 作成
- `request.md` を読み、**目標を一意に決めるための情報が足りているか**を主エージェント自身で検討する。
  足りているかを判断するのは以下の観点:
  - 何を実現したいかが具体的か(変更対象・期待する動作が読み取れるか)
  - スコープの境界が引けるか(非目標が想像できるか)
  - 既存規約・配布方針との衝突が予想されるか
- **不足があれば利用者に問い合わせる**(短く・要点だけ)。回答を受けたら反映する。
- 情報が揃ったら **`checkedRequest.md`** を `docs/design/<branch>/checkedRequest.md` に作成。
  項目:
  - **目標**(5〜10 行)
  - **スコープ / 非目標**
  - **規約参照**(リンクのみ。内容のコピー貼付はしない)
  - **起点ヒント**(あれば関連パス1〜2行)
  - **想定される影響範囲**(主エージェントが規約・記憶から見えている範囲のみ。コードは読まない)
- **コミット**: `git add docs/design/<branch>/checkedRequest.md && git commit -m "docs: checkedRequest 作成"`。

### 1. planning フェーズ
worker をフェーズ `planning` で起動する。
- 入力で渡すもの:
  - **フェーズ**: `planning`
  - **対象 design ディレクトリ名**: `<branch-name>`
  - **(任意)起点ヒント**
- worker は `request.md` / `checkedRequest.md` を自分で読む。主エージェントは要求文を貼らない。
- 戻り値: `Plan.md` が作られていること、worker 自身でコミット済みであること。

### 2. 設計レビューのラリー(最大2往復)
1. **designReviewer を起動**: 入力 = 対象 design ディレクトリ名と「Plan を確認」の指示。
2. `designReview.md` の最新巡を読み、判定を確認:
   - `GO` → §3 へ
   - `要相談` → **クリティカル停止**(後述)。特に観点 0 で上流(request / checkedRequest)の問題が指摘されていれば、修正して再度 designReviewer を起動する判断もある。
   - 指摘あり(判定がまだ無いか「GO」) → 3 へ
3. **worker をフェーズ `rally-response` で起動**。
4. **designReviewer を再起動**(2巡目)。判定が:
   - `GO` → §3 へ
   - 重大未解決あり / `要相談` → **クリティカル停止**

### 3. 実装フェーズ
worker をフェーズ `implement` で起動する。
- 入力: 対象 design ディレクトリ名のみで足りる(worker は Plan.md / request.md / checkedRequest.md を読む)。
- worker は src/tests を編集し、red → green を最大3周回し、自分でコミットする。
- 戻り値で次のいずれかが返ったらクリティカル停止:
  - **設計差し戻しシグナル**
  - 3周で緑にならない / 環境起因の flaky

### 4. 最終レビューのラリー(最大2往復)
1. **reviewer を起動**: 入力 = 対象 design ディレクトリ名、変更範囲のヒント(任意)。
2. `finalReview.md` の最新巡を読み、判定を確認:
   - `GO` → §5 へ
   - `要相談` → **クリティカル停止**
   - 指摘あり → 3 へ
3. **worker をフェーズ `review-fix` で起動**。worker は応答記入+修正+テスト緑+コミット。
4. **reviewer を再起動**(2巡目)。判定が:
   - `GO` → §5 へ
   - 重大未解決あり / `要相談` → **クリティカル停止**

### 5. 仕上げ
- 各エージェントが既にフェーズごとにコミットしているので、最後の追加コミットは原則不要。
- 念のため `git status` でクリーンを確認(コミット漏れがあれば主エージェント側で適切な prefix でコミット)。
- **マージはしない**(マージは利用者の明示指示があってから)。
- リモート操作はしない。
- 利用者には「何ができるようになったか/何が変わったか」を中心に簡潔に報告する(コミット履歴の列挙は不要)。

## クリティカル停止の手順
クリティカル停止が必要になったら:
1. 主エージェントの応答で次を伝える:
   - 何のフェーズ(設計レビュー / 設計差し戻し / 最終レビュー / 上流ドキュメント問題 / その他)
   - 関連ファイルへのリンク(`request.md` / `checkedRequest.md` / `Plan.md` / `designReview.md` / `finalReview.md`)
   - 解決できなかったことの 3〜5 行要約
2. 利用者の判断を待つ。
3. 判断を受け取ったら、必要なエージェントを再起動して続行する。

## オーケストレーションの原則(再掲)
- **重要なトレードオフ以外は主エージェントが解決**する(クリティカル停止は最小限)。
- サブエージェント同士は直接やり取りしない。**連携はすべて主エージェント経由**。
- 主エージェントはコードベースを読まない(原則)。読みは worker に集約する。
- 要求は `request.md` / `checkedRequest.md` という**ファイルで渡す**。チャットに長文を貼らない。
- 各サブエージェントが起動時にメモリを点検し、終了時に教訓を追記する規律を持つ。古い記述は削除・修正してから新規追記。
