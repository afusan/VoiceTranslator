リクエスト: $ARGUMENTS

あなたは主エージェント(オーケストレータ)として、上のリクエストを **自走** で実装まで回せ。
運用は `docs/design/AgenticWorkflow.html` の規律に従う。**原則「クリティカル時以外は止まらない」**。

## 手順

進捗を `TaskCreate` で起こし、各フェーズを `in_progress` / `completed` で更新しながら進める。

### 0. 目標(WHAT)と作業ブランチを決める
- 利用者リクエストを **CLAUDE.md・直近会話・自分の記憶だけで** 5〜10 行に言い直す。
  この段では **コードベースを読まない**(二重読込を避けるため、探索は worker に任せる)。
- 現在のブランチを確認:
  - `master` / `main` にいる場合のみ、内容を表す名前で新ブランチを切る
    (`feature/` / `bugfix/` / `refactor/` / `docs/` / `test/` / `chore/` のいずれか)。
  - 既に作業ブランチにいるなら、その上で続行(対象 design ディレクトリ名はブランチ名と一致させる)。
- 利用者には目標を **応答に短く示すだけ**(止まって確認は取らない)。

### 1. planning フェーズ
worker をフェーズ `planning` で起動する。
- 入力で渡すもの:
  - **フェーズ**: `planning`
  - **目標**: 上で起こした 5〜10 行
  - **対象 design ディレクトリ名**: `<branch-name>`
  - **規約参照**: `CLAUDE.md`, `docs/design/AgenticWorkflow.html`, 関連既存設計のリンクのみ
    (内容のコピー貼付はしない)
  - **起点ヒント**: あればパス1〜2行(無ければ省略)
  - **非目標**: あれば短く(無ければ省略)
- 戻り値: `docs/design/<対象>/Plan.md` が作られていること

### 2. 設計レビューのラリー(最大2往復)
1. **designReviewer を起動**: 入力 = 対象 design ディレクトリ名と「Plan を確認」の指示。
2. `docs/design/<対象>/designReview.md` の最新巡を読み、判定を確認:
   - `GO` → §3 へ
   - `要相談` → **クリティカル停止**(後述の手順へ)
   - 指摘あり(判定はまだ無いか「GO」) → 3 へ
3. **worker をフェーズ `rally-response` で起動**: 入力 = 対象 design ディレクトリ名と「designReview.md の指摘に応答」の指示。
4. **designReviewer を再起動**(2巡目)。判定が:
   - `GO` → §3 へ
   - 重大未解決あり / `要相談` → **クリティカル停止**

### 3. 実装フェーズ
worker をフェーズ `implement` で起動する。
- 入力: 対象 design ディレクトリ名のみで足りる(worker は Plan.md を読む)。
- worker は src/tests を編集し、red → green を最大3周回す。
- 戻り値で次のいずれかが返ったらクリティカル停止:
  - **設計差し戻しシグナル**(I/F・役割・スコープに及ぶ変更が必要)
  - 3周回しても緑にならない / 環境起因の flaky

### 4. 最終レビューのラリー(最大2往復)
1. **reviewer を起動**: 入力 = 対象 design ディレクトリ名、目標(再掲)、変更範囲のヒント。
2. `docs/design/<対象>/finalReview.md` の最新巡を読み、判定を確認:
   - `GO` → §5 へ
   - `要相談` → **クリティカル停止**
   - 指摘あり → 3 へ
3. **worker をフェーズ `review-fix` で起動**: 入力 = 対象 design ディレクトリ名と「finalReview.md の指摘に応答+修正」の指示。worker は応答記入と修正、テスト緑まで回す。
4. **reviewer を再起動**(2巡目)。判定が:
   - `GO` → §5 へ
   - 重大未解決あり / `要相談` → **クリティカル停止**

### 5. コミット
- 全テスト緑を最終確認(`py -m uv run pytest -q` でサマリ行を見るだけで足りる)。
- `git status` で変更ファイルを確認し、関連ファイルを **明示的に** `git add` する
  (`git add -A` / `.` は使わない — CLAUDE.md 方針)。
- コミットメッセージ: prefix を CLAUDE.md の対応表(`feat:` / `fix:` / `refactor:` / `docs:` / `test:` / `chore:`)から選び、HEREDOC で渡す。
- **マージはしない**(マージは利用者の明示指示があってから)。
- リモート操作はしない。
- 利用者には「何ができるようになったか/何が変わったか」を中心に簡潔に報告する。

## クリティカル停止の手順
クリティカル停止が必要になったら:
1. 主エージェントの応答で次を伝える:
   - 何のフェーズ(設計レビュー / 設計差し戻し / 最終レビュー / その他)
   - 関連ファイルへのリンク(`Plan.md` / `designReview.md` / `finalReview.md`)
   - 解決できなかったことの 3〜5 行要約
2. 利用者の判断を待つ。
3. 判断を受け取ったら、必要なエージェントを再起動して続行する。

## オーケストレーションの原則(再掲)
- **重要なトレードオフ以外は主エージェントが解決**する(クリティカル停止は最小限)。
- サブエージェント同士は直接やり取りしない。**連携はすべて主エージェント経由**。
- 主エージェントはコードベースを読まない(原則)。読みは worker に集約する。
- 規約参照は **リンクのみ**渡す(内容コピーで二重化を増やさない)。
- サブエージェントの最終行動として **メモリへの教訓追記** が走っているか、戻り値で軽く確認する(無ければ次回からの改善余地)。
