# worker メモリ

実装作業で得た**再現性のある教訓**を蓄積する。
具体(ファイル名・行・個別の修正)ではなく、繰り返す実装上の落とし穴・プロジェクト/利用者の
好み・有効だったテスト手順・配布方針との衝突の見抜き方を、**抽象度を保って**記す。

## 教訓

- i18n カタログの削除は 4 言語(ja/en/zh/es)を必ず同時に行う。`test_catalog_key_parity` がロケール間の整合をチェックするため、1 ロケールだけ消すと即座に赤になる。
- ConfigStore の DEFAULT_CONFIG からキーを消すとき、そのキーしか持たない backend ブロックは `{}` として残す(将来の追加先を保持)。
- planning フェーズで「消すだけ」のリファクタをする場合、削除順は「一番末端の実装(i18n キー) → スキーマ宣言 → ロジック → 呼び出し元と被呼び出しを同時 → テスト → ドキュメント」の順が壊れにくい。呼び出し元(E)と被呼び出し(F)は同時削除しないと中間状態で AttributeError が出るリスクがある。
- 「消すだけ」変更で構文エラーが出る場合、削除したエントリの直後に元から壊れていた行が隠れていたケース(全角クォート等)を疑う。削除後に露出することがある。
- 複数のメソッドが同一 widget の state を書き換える場合、「最後に呼んだ方が勝つ」ため呼び出し順序が正しさを決める。Plan で「干渉しない」と書かれていても、is_none=False 時に "normal" に戻す処理が has_settings で設定した "disabled" を消す例があった。担当の明確化(あるメソッドが担当する条件は限定し、それ以外は触れない)と末尾の適用順固定の両方を合わせて初めて安全になる。
