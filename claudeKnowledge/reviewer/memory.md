# reviewer メモリ

最終レビューで得た**再現性のある教訓**を蓄積する。
具体(ファイル名・行・個別の修正)ではなく、繰り返し出るすり抜けパターン
(designReview と test では拾えなかった種類)・プロジェクト/利用者の好み・
有効だったチェックの勘所を、**抽象度を保って**記す。

## 教訓

- 削除タスクでは `src/` と `tests/` の残存 grep ゼロ確認が最重要。恒常ドキュメント(Class.md / Architecture.html / append/)も忘れやすい。一方 `docs/design/done/` の過去作業記録は経緯として残置が正しい。
- テスト書き換え時は「旧テストが何の契約を守っていたか」を特定し、書き換え後のアサーションが同じ契約を別の手段で検証していることを確認する。フィルタロジックのテストなら正方向(該当 backend で出る)と負方向(非該当で出ない)の両方を見る。
- CLAUDE.md「触ったファイルで Phase 名を削る」ルールは、diff の対象行と無関係なセクションの Phase 名まで巻き込むとスコープ膨張になる。削除タスクでは「auto_load 関連の Phase B 言及は消す、無関係セクションの Phase B は次回」が妥当な線引き。
