# voice-translator 使い方マニュアル

ローカル PC 上で動く音声翻訳アプリの使い方。インストールから日常の操作、
チューニング、トラブルシュートまでを扱う。設計情報は [docs/design/](design/) を参照。

---

## 1. このアプリでできること

- **PC 内で再生されている音声**(YouTube / Twitch / 通話など)を取り込み、別言語に翻訳した
  音声を**別の出力デバイス**から再生する。
- **特定アプリの音だけ**を取り込む(per-process キャプチャ、Windows)。
- **自分のマイクの声**を翻訳して、スピーカや仮想マイク(通話アプリへの入力)から再生する。
- **テキスト字幕モード**: 翻訳結果を画面表示のみで完了する構成(TTS / 出力デバイス不使用)。

処理の流れ(各段は独立したスレッドで動作):

```
[入力デバイス] → VAD → ASR → 翻訳 → TTS → [出力デバイス]
                (発話区切り) (書き起こし) (翻訳) (音声合成)

# テキスト字幕モード(TTS=「(なし)」)
[入力デバイス] → VAD → ASR → 翻訳 → 履歴表示で完了(TTS / Output は起動しない)
```

すべての処理は**ローカルで完結**する(既定構成)。クラウド backend を選んだ場合のみ、
該当レイヤの処理が外部 API に送信される。

---

## 2. 動作環境

- **OS**: Windows 11(動作確認の主対象)。macOS / Linux は一部 backend を除き動作:

  | Windows 専用 | 影響 | 他 OS での代替 |
  |---|---|---|
  | `sapi` TTS(既定 TTS) | 読み上げ不可 | `piper` / クラウド TTS / TTS=「(なし)」(字幕モード) |
  | `proc-tap`(per-process キャプチャ) | 特定アプリの音だけ翻訳が不可 | 代替なし |
  | スピーカループバック(`[LB] ...`) | OS ごとに仕組みが違う | macOS: BlackHole 等 / Linux: PipeWire monitor を別途用意 |

  VAD / ASR / 翻訳 / Piper TTS / MMS-TTS / クラウド系 backend はクロス OS で動作する。
- **Python**: 3.12 以上(`uv` が自動取得する。手動インストール不要)
- **ディスク**: 5〜6GB 程度(モデル + PyTorch 等。extras を足すと増える)
- **メモリ**: 8GB 以上推奨(翻訳モデル NLLB-200 600M を CPU で動かすため)
- **GPU**: 任意。NVIDIA GPU があれば自動で使われ、ASR / 翻訳が大幅に速くなる
- **ネット**: 初回起動時のモデルダウンロードのみ必要(以降はオフライン動作可)

## 3. インストール

`uv`(Python のモダンな環境管理ツール)を使う。

```bash
# 1) uv を入れる(いずれか 1 つ)
py -m pip install --user uv
# or: powershell -c "irm https://astral.sh/uv/install.ps1 | iex"

# 2) 依存と Python を自動セットアップ(--extra cpu / cuda はどちらか必須)
py -m uv sync --extra cpu          # CPU 専用(誰でも動く、推奨初手)
# py -m uv sync --extra cuda       # NVIDIA GPU 向け(自動で GPU 使用、+約3GB)

# 追加 backend が欲しい場合(任意):
# py -m uv sync --extra cpu --extra capture-proctap   # 個別追加の例(ProcTap)
# py -m uv sync --extra cpu  --extra full              # 全部のせ(CPU)
# py -m uv sync --extra cuda --extra full              # 全部のせ(CUDA)
```

### `--extra cpu` と `--extra cuda` の選び方

| 環境 | 推奨 | 備考 |
|---|---|---|
| Windows / Linux(NVIDIA GPU 無し) | `--extra cpu` | |
| Windows / Linux + NVIDIA GPU(RTX 等) | `--extra cuda` | ASR / 翻訳が 5〜15 倍速 |
| macOS(Apple Silicon) | `--extra cpu` | MPS が自動で使われる(翻訳のみ) |
| その他(Intel Mac / AMD GPU) | `--extra cpu` | |

- 2 つは**排他**(同時指定はエラー)。**CUDA Toolkit のインストールは不要**
  (wheel に同梱。NVIDIA ドライバがあれば動く。`nvidia-smi` で確認)。
- 後から切り替える場合は `uv sync --extra <別の方>` を再実行(差分のみ更新)。

### 追加 backend(extras)

追加の VAD / クラウド ASR・翻訳・TTS / Piper / ProcTap は **extras を入れた分だけ**
GUI のプルダウンに現れる(未導入の backend は列挙されない)。
全部入れる集約プリセットが `--extra full`。個別に選びたい場合の対応表は
`pyproject.toml` の `[project.optional-dependencies]` と README の backend 一覧を参照。

### **重要**: 起動コマンドにも同じ `--extra` を付ける

`uv run` は既定で「extras 無し」の sync を再実行するため、起動時に `--extra` を省くと
**せっかく入れた extras が剥がされる**(CUDA 版 torch も CPU 版に戻される)。

```bash
py -m uv run --extra cuda --extra full python -m voice_translator   # 例: CUDA 全部のせ
py -m uv run --extra cpu python -m voice_translator                 # 例: CPU 最小
```

「今の venv のまま実行したい」場合は `--no-sync`:
```bash
py -m uv run --no-sync python -m voice_translator
```

## 4. 起動

```bash
py -m uv run --extra cpu python -m voice_translator   # sync 時と同じ extras を付ける
```

GUI ウィンドウが立ち上がる。

> **モデルロードのタイミング**: 既定では起動時にモデルはロードされない(各レイヤ `Init`)。
> ロードは **「▶ 開始」押下時に未ロード分がまとめて**、または **「↻ ロード」ボタン**で行われる。
> 一度ロードしたモデルは停止後も常駐するため、Stop → Start の繰り返しでロード待ちは発生しない。
> 特定 backend を起動時に先行ロードしたい場合は、レイヤの「設定」ダイアログで
> 「起動時に自動ロード」を ON にする。
>
> **初回起動の注意**: 初回は ASR モデル(数百 MB)と翻訳モデル(NLLB-200 約 2.5GB)の
> ダウンロードが走る(数分〜十数分)。2 回目以降はキャッシュから読み込まれる。

## 5. 画面と初期設定

設定パネルは「**バックエンド**」「**デバイス**」「**翻訳**」の 3 セクション
(見出しクリックで個別に折り畳み可、状態は永続化)。下部に「ログ出力先」
「設定を保存 / 再読込 / デバイス再列挙」ボタンが常時表示される。

### 5-1. デバイスの選び方

- **入力デバイス**:
  - PC 音声を翻訳したい → `[LB] スピーカ名`(ループバック)を選ぶ
  - 自分の声を翻訳したい → 通常のマイクを選ぶ
  - 一覧は「音声取得」backend に連動する(`soundcard` ならデバイス一覧、
    `proctap` なら「プロセス選択…」ボタンに切り替わる)
- **出力デバイス**: **必ず入力とは別のデバイス**を選ぶ。同じだと翻訳音声を再キャプチャ
  する無限ループになるため、起動時にエラーで弾かれる。
- 例(英語 YouTube → 日本語音声): 入力 `[LB] Speakers` / 出力 `Headphones`

**動作中のデバイス切替**: 動作中に入出力デバイスを変更すると**自動で停止 → 再開**して
切り替わる(1〜2 秒の中断。バナーで進行表示、失敗時は赤バナー + 停止)。

### 5-2. 翻訳言語

- **入力言語 (src)**: 翻訳元。`auto` で自動検出可。候補は選択中の ASR backend の対応言語に連動。
- **出力言語 (tgt)**: 翻訳先。候補は「**翻訳が出せる言語 ∩ TTS が読み上げられる言語**」
  (TTS 無効時や対応言語不明の TTS では絞らない)。翻訳 / TTS を切り替えると候補も連動し、
  現在の選択が外れた場合は自動で切り替わる(バナーで明示)。
- **言語コード表記**: 内部は ISO 639-3(例: `eng (English)` / `jpn (Japanese)` / `swh (Swahili)`)。
- **🔍 検索**: 各言語行の「🔍」ボタンで検索ダイアログを開ける。`mms` TTS を選ぶと候補が
  100 言語超になるため、コードや英語名で絞り込んで選べる。
- **動作中の変更可**: 次の発話から新言語が適用される(停止不要。キュー内の発話は旧言語で完走)。

### 5-3. バックエンド選択

各レイヤ(音声取得 / VAD / ASR / 翻訳 / TTS / 音声出力)の実装をプルダウンで選ぶ。

- プルダウンには**依存パッケージが導入済みの backend だけ**が並ぶ(§3 の extras 参照)。
- 切り替えるとそのレイヤだけ `Init` に戻る(破棄のみ。自動再ロードはしないので、
  押し間違えてもすぐ選び直せる)。実ロードは次の「▶ 開始」/「↻ ロード」。
- **動作中はプルダウンと「設定」ボタンが無効化**される(表示と実際に動いているものの
  食い違い防止)。停止すると解除される。デバイスと言語は動作中も変更可。

**ステータスラベル**(各行右側、英語・色付き):

| 表示 | 色 | 意味 |
|---|---|---|
| `Init` | グレー | 初期状態(まだロードしていない) |
| `Missing Credentials` | 赤 | クラウド backend の認証情報が未入力(§5-7) |
| `Not Verified` | オレンジ | 認証情報は保存済みだが疎通テスト未実施(§5-7) |
| `Not Downloaded` | 赤 | ロード失敗(依存不足 / モデル DL 失敗等。§9-4) |
| `Downloading...` | オレンジ | モデルをダウンロード中 |
| `Loading...` | オレンジ | メモリへロード中 |
| `Loaded (cuda)` 等 | 緑 | ロード済み(ASR / 翻訳は使用デバイスを併記) |

`Missing Credentials` / `Not Verified` は**ロードしなくても**認証情報から判定して表示され、
解消するまで「▶ 開始」が無効化される。

### 5-4. ASR+翻訳の一括バックエンド(複合)

ASR に以下を選ぶと、書き起こしと翻訳を 1 回の処理で行う(End-to-End 音声翻訳):

| 名前 | 形態 | 翻訳先 | 原文表示 | 備考 |
|---|---|---|---|---|
| `faster_whisper_translate` | ローカル(無料) | 英語固定 | なし | Whisper の translate タスク。NLLB のロード(約 2.5GB)が不要になり軽い |
| `openai_whisper_api_translate` | クラウド(要 API key) | 英語固定 | なし | OpenAI Whisper API の translations。25MB/発話の上限 |
| `gpt_audio_translate` | クラウド(要 API key) | **任意** | **あり** | GPT 音声入力モデル。既定は mini 系(設定で変更可) |

共通のふるまい:
- **翻訳レイヤは使われなくなる**(ロードもされない)。翻訳行のプルダウンと設定ボタンは
  disabled、ステータス欄は空表示になる。選択値は記憶されており、ASR を単体 backend に
  戻すと翻訳レイヤが復帰する。どの backend が代行しているかは動作タブのステータス集約で
  確認できる(例: `[translator] (asr の faster_whisper_translate で実行)`)。
- 出力言語の候補は複合 backend が決める(英語固定の 2 つでは `eng` のみ)。
- クラウド 2 種の API key 未登録時は、同じ OpenAI key を使う既存 backend
  (`openai_whisper_api` / `openai_gpt`)の保存済み key を自動で使い回す。

### 5-4b. 多言語読み上げ(MMS-TTS)

低資源言語(アフリカ系等)も読み上げたい場合は TTS に `mms`(Meta MMS-TTS)を選ぶ。

- **対応言語**: 翻訳(NLLB-200)と読み上げ(MMS)の両方が成立する **99 言語**。スワヒリ・
  ヨルバ・ハウサ・アムハラ・チェワ・ガンダ等、`sapi` / `piper` が読めない言語を多く含む。
- **主要言語の非対応に注意**: MMS は低資源言語向けのため、**日本語(jpn)・中国語(zho)・
  アラビア語(ara)・イタリア語(ita)などは読み上げできない**(チェックポイントが存在しない)。
  これらを出力したいときは `sapi`(Windows、日本語あり)/ クラウド TTS(OpenAI / Google /
  ElevenLabs)を使う。`mms` を選ぶと出力言語候補からこれらが消え、現在値が日本語等なら英語へ
  自動 fallback する(バナーで明示)。**日本語で喋らせたい用途では `mms` 以外を選ぶこと。**
- **言語パックは遅延ロード**: 言語ごとに小さなモデル(~100〜150MB)を初回利用時に自動
  ダウンロードする。出力言語を選んだ時点で裏で先読みするので、会話中の初回発話が固まりにくい。
- **導入**: エンジン本体は標準依存に含まれる。一部言語の前処理(ローマ字化)に必要な追加分のみ
  `uv sync --extra tts-mms`。
- **ライセンス注意**: MMS のモデルは **CC-BY-NC 4.0(非商用)**。NLLB-200 と同じく、商用利用では
  別 backend(クラウド TTS 等)に切り替えること。

### 5-5. テキスト字幕モード(TTS=「(なし)」)

TTS プルダウン末尾の `(なし)` を選ぶと:
- TTS / Output レイヤを使わない構成になり、翻訳結果は**画面の履歴表示のみ**。
- 「音声出力」行はグレーアウト(選択値は記憶され、復帰時に戻る)。
- TTS / Output のロード・認証も不要になる(クラウド TTS の認証なしでも開始できる)。
- 切替は**次回 Start で反映**(動作中なら Stop → Start)。

### 5-6. プロセス単位キャプチャ(ProcTap、Windows)

音声取得に `プロセス (proctap)` を選ぶと(要 `--extra capture-proctap`)、特定アプリの
音だけを取り込める。「入力デバイス」行が「**プロセス選択…**」ボタンに切り替わる:
- 音声出力中のプロセス一覧から選択。「▶ 試聴開始」でレベルメータ確認、「↻ 更新」で再列挙
- 選択した PID は**保存されない**(プロセス再起動で PID が変わり、別アプリの音を取り込む
  事故を防ぐため。起動のたびに選び直す)
- 未選択のまま開始しようとするとボタンが「プロセス未選択」になり阻止される
- **対象アプリの音量が小さくて認識されない場合**: 音声取得行の「設定」→
  「ProcTap: 入力ゲイン (倍率)」で内部増幅できる(2〜8 倍が目安。アプリ側音量が 0 だと
  増幅不能なので最低限は上げておく)。変更後は「モデルを(再)ロード」で反映

### 5-7. クラウド backend の認証

クラウド backend(`openai_*` / `deepl` / `google_*` / `deepgram` / `elevenlabs` /
`anthropic_claude` や、認証必須のローカル backend `pyannote` / `pvcobra`)を使う手順:

1. プルダウンで選択すると**利用同意ダイアログ**が出る(サービスの規約 URL を表示。
   同意しないと選択はキャンセルされる)
2. 行ステータスが `Missing Credentials`(赤)になる → レイヤ行の「**設定**」→「**認証**」
3. API key 等を入力して「**テスト**」を押す(実際に疎通確認する)
4. 成功すると保存され、ステータスの認証表示が解消 → 「▶ 開始」が押せるようになる

- key を入力しただけでテストを通していない状態は `Not Verified`(オレンジ)で、開始は
  ブロックされたまま。**必ず「テスト」まで通す**。
- 実行中に認証エラー(401 等)を検知すると verified が失効し、再テストが必要になる。
- 保管先は **OS の資格情報ストア**(Windows Credential Manager / macOS Keychain)。
  使えない環境ではプロジェクト直下の `local.secrets` に平文 fallback(git 管理外)。

### 5-8. レイヤ別の詳細設定(「設定」ボタン)

各レイヤ行の「設定」ボタンで、そのレイヤ固有の設定ダイアログが開く。代表的な項目:

- **全 backend 共通**: 「起動時に自動ロード」トグル / 「モデルを(再)ロード」ボタン /
  直近の処理時間表示 / (クラウド系)「認証」ボタン
- **ASR**: Whisper モデルサイズ(tiny〜large、dropdown)、クラウド系のモデル名
- **翻訳**: NLLB-200 のモデル(600M〜3.3B)、LLM 翻訳のモデル名
- **TTS**: SAPI 読み上げ速度 / Piper voice / ElevenLabs voice 等
- **VAD**: WebRTC 感度・フレーム長 / pyannote モデル / Cobra 閾値 等
- **バッファ容量**: 各ステージ間キューの上限(§9 チューニング参照)

モデル等を変更したら「モデルを(再)ロード」で反映する(または次の開始時)。
永続化したい場合は設定パネル下部の「設定を保存」。

## 6. 動作開始 / 停止

- **「▶ 開始」**: 未ロードのモデルがあればその場でまとめてロードしてから起動する。
  事前に「↻ ロード」済みならほぼ即座に始まる。無効化されるのは
  「認証情報未設定」「認証未検証」「モデル DL 中…」「プロセス未選択」のときだけ。
- **「■ 停止」**: 処理スレッドのみ停止。モデルは常駐したままなので再開はロード待ちなし。
- 動作中は翻訳テキスト履歴(「クリア」で消去可)と**平均レイテンシ**
  (発話終端の確定 → 再生指示まで、直近 10 件平均)が表示される。
- **致命的エラー**時は履歴に `[致命的エラー]` が出て「停止中(エラー)」になる。
  軽微な警告は UI に出さず `app.log` のみ。
- 「演算: GPU (cuda)」「演算: CPU のみ」の集約表示で、実際に GPU が使われているか確認できる。

### 「🔊 出力テスト」ボタン(切り分け用)

選択中の TTS / 出力デバイスで「テスト音声」と 1 回だけ読み上げる。本体を動かさずに
「TTS → 出力デバイス → スピーカ」だけを叩くので、「翻訳までは出るのに音が鳴らない」の
切り分けに使う。TTS=(なし) / 出力未選択 / 動作中は無効化される。

## 7. 設定の保存と読込

- **「設定を保存」**: 現在の設定を `config.yaml`(カレントディレクトリ)へ書き出す。
  ※ ProcTap の選択 PID はファイルには書かれない(セッション中の選択は維持される)。
- **「設定を再読込」**: ファイルから読み直す。**ファイルと実効内容が異なるレイヤだけ**が
  `Init` に戻り(他はロード済み維持)、次の開始で反映される。動作中・ロード中は実行不可。
- 次回起動時は保存済み `config.yaml` が自動で読み込まれる。

### config.yaml を手で編集した場合

`config.yaml` は起動時に 1 回だけ読み込まれる(ファイル監視はしない):

| やりたいこと | 手順 |
|---|---|
| 起動前に編集した | 通常起動するだけで反映 |
| 起動中に編集した | 「設定を再読込」→ 次の「▶ 開始」で反映 |
| GUI で変更した内容を残したい | 「設定を保存」(ファイルに書き戻し) |

- 「設定を保存」は in-memory の状態でファイルを上書きする。手編集と GUI 操作を混ぜると
  上書き事故が起こり得るので、どちらかに統一するのが安全。
- `pipeline.*` の値は「▶ 開始」時に読み込まれる(動作中の変更は次の開始から)。
- 書いていないキーは既定値が自動補完されるので、不要なキーは省略してよい。

## 8. ログ・履歴

| ファイル | 内容 |
|---|---|
| `<ログ出力先>/app.log` | アプリ全般ログ |
| `<ログ出力先>/translations.jsonl` | 翻訳履歴(1 行 = 1 発話。ts / seq_id / src・tgt テキスト / 言語 / latency_ms / timeline) |
| `<ログ出力先>/soundsrc.txt` | (任意)翻訳前テキスト。既定 OFF |
| `<ログ出力先>/translated.txt` | (任意)翻訳後テキスト。既定 OFF |
| `<ログ出力先>/processtime.csv` | (任意)レイヤ別処理時間。既定 OFF(§9) |

発話ごとに一意な連番 `seq_id` が全ログに載るので、`grep "#42"` や
`jq 'select(.seq_id == 42)'` で「どの段まで到達したか / どこで落ちたか」を突き合わせられる。

```yaml
log:
  directory: ./logs
  jsonl_enabled: true
  src_text_enabled: true     # soundsrc.txt を出力
  tgt_text_enabled: true     # translated.txt を出力
  level: INFO                # WARNING に上げると SKIP(無音等)のノイズを抑制
notifications:
  throttle_sec: 5.0          # 同種エラーの UI 通知を 5 秒に 1 回へ集約(0 で無効)
```

- テキストログは追記モード(ローテーションなし)。キューあふれで再生されなかった発話も
  src/tgt は記録される(各段で直接書かれるため)。
- 通知の抑制件数は次の通知に `(+N件抑制)` と表示。app.log には全件残る。

## 9. チューニング

### GPU(CUDA / Apple Silicon)

NLLB-200(翻訳)と faster-whisper(ASR)は GPU があれば**自動で**使う(`device: auto` 既定。
CUDA → MPS → CPU の順)。状態は各レイヤの `Loaded (cuda)` 表示と「演算:」集約表示で確認。
GPU 初期化に失敗した場合は自動で CPU にフォールバックする(ログに記録)。

```yaml
backends_config:
  nllb200:
    device: auto          # "auto" / "cuda" / "mps" / "cpu"
  faster_whisper:
    device: auto          # "auto" / "cuda" / "cpu"(CTranslate2 は MPS 未対応)
    compute_type: auto
```

### レイテンシを下げる

- **ASR モデルを小さくする**: ASR 行の「設定」→「Whisper モデル」で `tiny` / `base` に
  (精度とのトレードオフ)。
- **複合 backend を使う**(§5-4): 英語への翻訳でよければ `faster_whisper_translate` で
  翻訳ステージごと省略できる。
- **GPU を使う**(上記)。
- ボトルネックの特定は `processtime.csv`(下記)で。

### 各レイヤの処理時間を CSV で記録(プロファイル)

```yaml
log:
  process_time_enabled: true   # ./logs/processtime.csv に追記
```

1 行 = 1 発話で、`asr_proc_ms` / `translate_proc_ms` / `tts_proc_ms` 等の**純処理時間**と、
`*_wait_ms`(直前キューでの待ち時間)、`total_ms`(端から端)が載る。
`*_proc_ms` が大きい段が処理ボトルネック、`*_wait_ms` が大きい段は直前の段が詰まっている。

### VAD の発話区切り(長文連続発話で詰まるとき)

ニュースのようにポーズの無い入力では 1 発話が巨大化して下流が破綻することがある。
Silero VAD の挙動を調整できる(VAD 行の「設定」または `config.yaml`):

```yaml
backends_config:
  silero:
    threshold: 0.5         # 判定しきい値(0〜1)
    min_silence_ms: 500    # 発話終了とみなす無音(短くすると早く区切れる)
    speech_pad_ms: 100     # 発話前後の余白
    max_speech_sec: 8.0    # 1 発話の最大長。超えたら強制区切り(0 で無効)
```

ノンストップ素材では `min_silence_ms: 200` + `max_speech_sec: 5.0` くらいが目安。

### ステージ間バッファ

```yaml
pipeline:
  captured_queue_max_bytes: 10000000     # 入力 → ASR(PCM、バイト基準。既定 10MB ≒ 156 秒)
  synthesized_queue_max_bytes: 5000000   # TTS → 出力(既定 5MB)
  recognized_queue_size: 10              # ASR → 翻訳(件数)
  translated_queue_size: 10              # 翻訳 → TTS(件数)
```

あふれたら古い発話から退避され、`app.log` に `queue overflow ...` で記録される。
翻訳 / TTS が詰まりやすいなら入力側を増やす、メモリを抑えたいなら下げる。

### TTS の読み上げ速度(SAPI)

```yaml
backends_config:
  sapi:
    rate: 220    # 既定 180。早口にすると再生時間が短くなりキューあふれ抑制に効く
```

### ① 海外配信を日本語音声で聞く
- 音声取得 `soundcard` / 入力 `[LB] Speakers` / 出力 `Headphones`
- 言語 src `auto` → tgt `jpn`

### ② 通話アプリの相手の声だけ翻訳する(Windows)
- 音声取得 `プロセス (proctap)` → 「プロセス選択…」で通話アプリを選ぶ
- 出力はヘッドホン等(通話アプリが使っていないデバイス)
- PC の他の音(BGM 等)が混ざらないのが利点

### ③ 自分の声を翻訳して通話相手に送る(仮想マイク)
- [VB-Audio Virtual Cable](https://vb-audio.com/Cable/) 等の仮想オーディオデバイスを別途インストール
- 本アプリ: 入力 = 実マイク / 出力 = `CABLE Input` / tgt = 相手の言語
- 通話アプリ(Teams / Discord 等)のマイク設定 = `CABLE Output`
- 自分の翻訳音声をスピーカーで聞くとマイクが拾ってループするので、モニタはヘッドホンで

### ④ 音を出さずに字幕だけ
- TTS = `(なし)`(§5-5)。翻訳結果は画面の履歴に流れる

## 11. トラブルシュート

### 「入力と出力に同じデバイスは使用できません」
フィードバックループ防止のため。別デバイスを選ぶ(§5-1)。

### バックエンドが `Not Downloaded`(赤)になる
ロード失敗の総称。主な原因と切り分け:
1. **依存パッケージ(extras)未導入** — 動作タブのステータス集約「最近のエラー」に
   `uv sync --extra xxx で追加してください` と出る。指示どおり extras を入れて再起動
   (なお通常は未導入 backend はプルダウンに出ないため、これが出るのは保存済み設定が
   未導入 backend を指している場合など)
2. **モデル DL 失敗** — 初回ロードはネット接続が必要。接続して再ロード
3. その他 — `app.log` に詳細が出る

### 「▶ 開始」が押せない
ボタンの表示が理由を示す:
- `認証情報未設定` / `認証未検証` → §5-7 の認証手順(「テスト」まで)を行う
- `モデル DL 中…` → ダウンロード完了を待つ
- `プロセス未選択` → ProcTap のプロセスを選ぶ(§5-6)

### 音声が出ない
- 「🔊 出力テスト」で TTS → 出力経路だけを確認(§6)
- 出力デバイスの選択 / OS のボリュームミキサーのミュートを確認
- 翻訳テキストは履歴に出ているか(出ていれば入力側は正常)

### 翻訳されない
- 「停止中(エラー)」になっていないか
- ProcTap の場合: 対象アプリの音量が小さすぎないか(§5-6 の入力ゲイン)
- `app.log` を確認

### レイテンシが大きい
§9「レイテンシを下げる」を参照(モデルサイズ / 複合 backend / GPU / processtime.csv)。

### モデルのキャッシュ場所
`~/.cache/huggingface/hub`(ASR / NLLB-200 とも)。一度ダウンロードすれば以降オフラインで動く。
