backend 候補(レイヤ別ざっくり比較)

黄色背景 = 現 MVP。ローカル / クラウド。 「必要リソース」は概算(GB)で、推論時のピーク。クラウドはネット帯域とレイテンシが事実上の要件。 「認証」は事前準備の重さ — 不要 / API key / OAuth 等。 Phase F でどれを試すかの叩き台として使う。

緑背景 + ✓ = 追加予定 — VAD/ASR/Translator から 3 つずつ選定(2026-05-29)。 選定方針: ローカル/クラウド両方を試せる多様性 + 認証フロー(Phase E-2)を実機で叩ける品揃え + 契約テスト雛形(`test_credential_flow.py`)と整合。

1. Capture(音声取得)

名前形態認証必要リソース備考
soundcard (現 MVP) ローカル 不要 ~50MB Windows/Linux で loopback ◯。Mac は BlackHole 別途。pip 1 つで完結
sounddevice (PortAudio) ローカル 不要 ~50MB クロス OS。loopback は OS 設定依存。numpy 親和性が高い
PyAudio ローカル 不要 ~50MB PortAudio wrapper。メンテ落ち気味。新規採用は避ける
WASAPI Process Loopback ローカル 不要 ~50MB Windows 10+ の Per-app loopback。実装は pycaw/自前 ctypes
ScreenCaptureKit (macOS) ローカル 不要(権限ダイアログ) ~50MB macOS 12.3+ の per-app audio capture。Python ラッパは限定的
仮想デバイス (VB-CABLE/BlackHole) ローカル 不要 ~50MB OS にドライバを足してルーティングする workaround。Mac の loopback 代替

2. VAD(発話区切り検出)

名前形態認証必要リソース備考追加
Silero VAD (現 MVP) ローカル 不要 RAM ~0.3GB / モデル ~7MB ONNX。pip パッケージ同梱、DL 不要。CPU で十分速い。日本語/英語実績あり —
WebRTC VAD ローカル 不要 RAM ~0.1GB C 実装。極軽量、ルールベース。長文発話の精度は Silero 劣後。選定理由: Silero がダメな環境(onnxruntime 不在等)のフォールバックとして、無認証ローカルの最軽量枠 ✓
pyannote.audio ローカル 不要(モデル DL) RAM ~2GB / VRAM ~1GB Transformer ベース。話者分離と一体運用可。重め。選定理由: 「重いがニューラル」軸でモデルサイズ選択(Phase C の hw_info 適合度)が機能するかの検証材料 ✓
Picovoice Cobra ローカル アクセスキー RAM ~0.1GB 個人非商用は無料 tier。商用は要ライセンス。極軽量。選定理由: 「ローカル動作だが認証が要る」型の検証材料(クラウドとは別パターン) ✓
energy-based 自作 ローカル 不要 ~0 RMS しきい値のみ。デバッグ/最終手段。精度低い —

openai-whisper (公式) ローカル 不要 RAM 2〜10GB / VRAM 2〜10GB PyTorch 直。faster-whisper より遅いが安定。研究/ベースライン向き —
whisper.cpp ローカル 不要 RAM 0.5〜5GB C++ 実装。Python 経由は ctypes / バインディング。組み込み向き —
vosk ローカル 不要 RAM 0.5〜2GB / モデル 50MB〜1GB Kaldi ベース。超軽量、低スペック PC で動く。精度は Whisper 劣後 —
OpenAI Whisper API クラウド API key ~50MB(HTTP client) $0.006/分。25MB/req 制限。レイテンシ実用域。large-v3 相当の精度。選定理由: 認証 1 つの最単純な cloud パターン、Phase F の本命候補。契約テスト雛形あり ✓
Deepgram (Nova-3) クラウド API key ~50MB $0.0058/分。WebSocket でリアルタイム。低レイテンシが売り。選定理由: ストリーミング ASR の検証材料(同期 transcribe との挙動差を確認) ✓
AssemblyAI クラウド API key ~50MB $0.37〜0.65/時間。話者分離・要約等の API も同居 —
Google Cloud STT クラウド サービスアカウント JSON ~100MB(SDK) $0.024/分(標準)。Chirp モデルでマルチ言語強い。選定理由: JSON ファイル形式の認証が要る(`file_picker` schema 型の追加検証込み)。GCP は別カテゴリの認証パターン ✓
Azure Speech クラウド サブスクリプションキー ~100MB(SDK) $1/時間相当。日本リージョン有り。継続発話で安価 —
AWS Transcribe クラウド IAM credentials ~100MB(boto3) $1.44/時間(標準)。IAM 設定が重め。ストリーミングあり —

Phase F 着手時の検討メモ

  • ASR で試すなら: OpenAI Whisper API(認証 1 つで動作確認しやすい、$ 安い)。Deepgram もリアルタイム性能の検証として面白い。
  • Translator で試すなら: DeepL API(日本語品質、無料 tier、認証 1 つ)。LLM 翻訳の比較対象として Claude Haiku もアリ。
  • TTS で試すなら: OpenAI TTS(認証 1 つ、自然、SAPI からの置換効果が明確)。VOICEVOX はローカル日本語の差し替えとして強い候補。
  • cloud backend は BackendRegistry.register(...., capabilities=BackendCapabilities(is_cloud=True, requires_credentials=True, service_name=..., terms_url=...)) で申告すれば、Phase D の同意ダイアログ + 認証フィールドが自動的に出る。

追加予定 backend サマリ(2026-05-29 選定)

レイヤ追加候補カバーする認証パターン
VAD WebRTC VAD / pyannote.audio / Picovoice Cobra 無認証ローカル軽量 / 無認証ローカル重量 / ローカル + キー認証
ASR OpenAI Whisper API / Deepgram / Google Cloud STT クラウド + 単一 API key / クラウド + 単一 API key(WebSocket) / クラウド + JSON ファイル
Translator DeepL API / OpenAI GPT-4o-mini / Anthropic Claude (Haiku) クラウド + 単一 API key × 3(LLM 翻訳の比較込み)

この組み合わせで以下が網羅できます:
(1) 認証パターン 3 種: 無認証 / 単一 API key / JSON ファイル
(2) 処理方式 2 種: REST 同期 / WebSocket ストリーミング
(3) 動作モデル 3 種: ローカル軽量 / ローカル neural / クラウド API
→ Phase E-2 の認証フローと Phase E のリトライ機構を実環境で叩く十分な多様性。
