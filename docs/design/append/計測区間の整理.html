レイテンシ計測区間の整理

本アプリは「マイクで声を拾う → 発話の区切り検出 → 文字起こし → 翻訳 → 音声合成 → 再生」を順番に流すパイプラインで、各処理の開始時刻と完了時刻を 1 発話ごとに記録している。GUI に出す「平均レイテンシ」がどこからどこまでを指しているか、どこに変更できるかを整理する。

1. 記録されている時刻(役割ベース)

マーカー意味(役割)
① 録音開始発話の先頭の音がマイクに入った瞬間(発話区切り検出器が「ここから発話」と判定した時刻)
② 発話終端確定発話区切り検出器が「ここで一区切り」と判定し、文字起こし待ち行列に投入した瞬間
③ 文字起こし開始待ち行列から取り出して、文字起こしエンジンに渡した瞬間
④ 文字起こし完了テキストと言語が確定した瞬間
⑤ 翻訳開始翻訳エンジンに渡した瞬間
⑥ 翻訳完了訳文が確定した瞬間
⑦ 音声合成開始音声合成エンジンに渡した瞬間
⑧ 音声合成完了再生用の音声データが揃った瞬間
⑨ 再生指示発行再生エンジンに「再生して」と指示した瞬間
⑩ 再生指示戻り再生エンジン側の関数が呼び出し元に制御を返した瞬間 (※ 実装次第で「再生キューに積み終わった」か「再生し終わった」かが変わる)

2. 時系列の図

発話そのもの   各段の待ち時間   各段の処理時間   再生

3. 今の GUI が出している区間

「① 録音開始 → ⑩ 再生戻り」。つまり「最初の音がマイクに入ってから、再生関数の戻りまで」の総時間を平均している。

この区間には発話そのものの長さがまるごと含まれる。たとえば 5 秒の文を喋ったら、内部処理がどんなに速くても表示は 5 秒以上になる。「翻訳が速いか遅いか」「会話の応答が速いか」を体感する指標としては分かりにくい。

4. 区間の候補

案区間意味用途注意
A(推奨) ② 発話終端 → ⑨ 再生指示 「喋り終わってから音が返ってくるまで」 会話の応答遅延を体感したい時。一般ユーザ向けの「平均レイテンシ」として最適 発話そのものの長さは含まれない
B ③ 文字起こし開始 → ⑨ 再生指示 「内部処理パイプラインだけの所要時間」(各段の待ち時間も含む) 開発・性能チューニング向け 体感とはやや遠い
C ① 録音開始 → ⑨ 再生指示 今の表示の「⑩ 戻り」を「⑨ 指示」に置き換えた版 今の表示の小改善 発話の長さが乗る問題は解決しない
D 段ごとに別々に表示(文字起こし / 翻訳 / 音声合成) 各段の処理時間を個別に出す どこが遅いか一目で分かる 1 つの数字にはならない / 表示スペース増
E ⑧ 音声合成完了 → ⑨ 再生指示 「処理は終わったのに音が出ない時間」(出力待ちのみ) 出力側の滞留デバッグ用 常時表示する値ではない

5. 推奨

GUI トップの「平均レイテンシ」は案 A(② 発話終端 → ⑨ 再生指示)に変更する。

必要なら、ステータスセクションに案 D(段別の経過時間)を追加表示すれば、「数字が悪いときにどこが原因か」も確認できる二段構成になる。

6. 補足: ⑩ 再生戻りの意味

再生エンジンの実装次第で ⑩ の意味が変わる:
  • 再生バッファに書いて即返るタイプ(典型的なオーディオ出力ライブラリ)→ ⑩ ≒ ⑨。レイテンシ表示には差が出ない
  • 再生完了まで待つタイプ → ⑩ には「再生時間そのもの」が乗る。会話被りの指標としては使えるが、応答遅延の指標としては不適
案 A は ⑨(再生指示) で止めることで、この実装差分の影響を受けないようにする狙いもある。
