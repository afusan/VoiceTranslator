AppController 責務マップ — MVVM 再構築版

作成: 2026-06-09。MVC 5 コラボレータ版(2026-06-05)に対する MVVM 再構築版。MVC と MVVM の違いを踏まえ、customtkinter の 制約下で実装可能な範囲に落とし込む。

1. 前提と方針

1-1. なぜ MVVM か

  • 現状の AppController は「Model + Controller(MVC)」を兼ねており、 GUI から呼ばれる Facade と内部ドメインの両方を抱えて 1,415 行に膨らんでいる。
  • 5 コラボレータ案(MVC 版)は責務分離は進むが、依然として「Controller が View を更新する」 命令型パターンを残す。SettingsPanel / ControlPanel の状態同期コードは消えない。
  • MVVM では View が ViewModel の Observable プロパティを購読 するため、 「設定変更したら別の widget が連動して更新される」式の手書き連動コードが消える。
  • 「設定の保存ボタン」のように MVC 文脈で常識化していた UI は、MVVM では ViewModel が Observable プロパティ書き換え時に自動永続化 する形に置き換える。

1-2. customtkinter での MVVM 実装の現実解 — 既存パターンの extraction で組む

MvcVsMvvm.html で確認した通り、customtkinter には公式の データバインディングが無い。**ただし**、本プロジェクトには既に Observable パターンの実体がある (src/voice_translator/common/backend_base.py の BackendBase.subscribe / Subscription)。 本計画は 「green-field で自前構築」ではなく、既存パターンを Observable[T] として extraction + 周辺部品(Command / TkBinder / DebouncedSaver)を薄く追加する。新規依存は入れない。 これにより:
  • プロジェクト中の既存スタイル(Subscription トークンによる購読解除、 listener-copy-on-notify、スレッドセーフ notify)と 1 つの流儀に揃う
  • traitlets / blinker のような 別パラダイムの併存を避ける (検討経緯は §8 末尾の代替案ノート参照)
  • コード量見積もりは ~200 行(本体)+ ~250 行(テスト)。 BackendBase の listener 機構を generic 化する extraction が中心
道具実装方針新規 / 既存
Observable[T] BackendBase の _listeners / _set_status / subscribe を generic に剥がしたもの。値変更時に listener を呼ぶだけ。common/observable.py ★ extraction(~40 行 + テスト ~50 行)
Subscription common/backend_base.py に既存。common/observable.py に移して BackendBase 側もこれを再利用する **既存を移設**(既存ファイル)
ObservableProperty(任意) descriptor 糖衣。vm.run_state.set(...) と書く煩わしさを許容するなら 不要。導入は後追いで OK 新規(~30 行)/ 後追いで判断

「既存パターンの extraction」の重み: BackendBase は既にスレッドセーフな Observable として動いている(_listeners_lock + listener コピー後のロック外通知、 Subscription トークン管理)。これを generic 化する作業は、新規構築ではなく 「特定型(ModelStatus 専用)を Observable[T] に汎用化する refactor」。 既存テスト(tests/test_backend_base.py 等)が保守される範囲で安全に進められる。
customtkinter の StringVar の限界: Entry や OptionMenu は textvariable 経由で「Entry の値 ↔ StringVar」 の双方向同期ができるが、CTkButton / CTkLabel の text は textvariable に対応しない widget もある。 StringVar が使えない widget は Observable.subscribe(lambda v: widget.configure(text=v)) で代替する(TkBinder が両ケースを吸収)。

1-3. 「設定の保存ボタン」を MVVM らしく置き換える

現状: ユーザが「設定を保存」を押すと ConfigStore.save() が呼ばれ config.yaml に書き出される。 MVVM 化の選択肢:
案挙動採否
A. Auto-persist(都度保存) ViewModel のプロパティが変わる度に Model 側で即 save。デバウンス(0.5 秒)で連続変更を集約 ★ 採用
B. Dirty flag + SaveCommand 変更があれば IsDirty=True、ボタンが enable に。押下で save 採用しない(操作が 1 つ増える)
C. Auto + Undo 都度保存 + 「設定を戻す」コマンドで 1 つ前にロールバック 保留(将来検討)

「設定を再読込」ボタンは MVVM でも維持する(外部編集の取り込み用)。 「設定を保存」ボタンは 撤去 し、デバウンス auto-persist で代替。 「デバイス再列挙」ボタンは Command として残す(ユーザの明示操作なので)。

4. データの流れ(イベント別)

イベント流れ
ユーザが ASR backend を変更 View(OptionMenu) → StringVar 変化 → BackendSelectionVM.select_backend() Command 発火 → ConfigStore 更新(auto-persist デバウンス起動) → BackendLoader.evict_layer → 状態 Observable が emit → LanguageSelectionVM の Reaction が src_choices を再計算 → View が自動更新
backend ロードが完了 BackendLoader が _set_status(LOADED) → backend.subscribe 経由でレイヤ別 Observable に通知 → BackendSelectionVM.statuses[layer] が更新 → ControlVM の Reaction が start_label / can_execute を再計算 → View が自動更新
ユーザが Start を押す View(button) → ControlVM.toggle_run AsyncCommand 起動 → DeviceValidator + 認証 gate → PipelineRunner.start_async → run_state を starting → running に遷移 → View が自動更新(「停止中」→「動作中」)
1 発話の翻訳が完了 PipelineRunner が ledger.pop + on_text_ready を UiDispatcher 経由で発火 → HistoryVM.entries に追加 → View が自動更新(履歴 widget)。 同時に HistoryVM.avg_latency_sec も再計算 → View 自動更新
設定の保存(MVC では「保存ボタン押下」) MVVM では 明示操作なし。ViewModel の Command で ConfigStore.set が呼ばれた時点で DebouncedSaver が 500ms タイマを起動 → タイマ満了で ConfigStore.save() 実行。 ユーザは「保存」を意識する必要がない。「設定を再読込」「デバイス再列挙」は明示 Command として残す
クラウド backend 選択 → 認証フロー BackendSelectionVM.select_backend → 同意 gate(ConsentVM dialog) → OK なら ConfigStore 更新 → 認証 gate(CredentialVM dialog) → verify → 保存 → verified Observable が True に → ControlVM の can_execute が再計算 → View 自動更新

5. MVC 案との対応関係

MVC 5 コラボレータ版と本 MVVM 案の対応:
MVC 5 コラボレータ案MVVM での置き場所備考
StatusBroadcaster(A+C) StatusVM + Model 側の状態 Observable UI に流す部分(VM)と、Model 内の通知部分(各 backend / loader)に分かれる
CredentialsCoordinator(D) Model: CredentialsCoordinator(同名で残す)+ VM: CredentialVM Model 側のロジックは MVC 案と同じ。VM は dialog 用
BackendLoaderService(F+E) Model: BackendLoader(rename) + メタ問合せ classmethod は各 BackendCls に残す VM は BackendSelectionVM / LanguageSelectionVM が直接 Backend クラスの classmethod を呼ぶ
SettingsFacade(B) Model: ConfigStore(Observable プロパティ化 or 変更通知付き) ConfigStore 自体に Observer を付け、ViewModel が購読
PipelineRunner(G+H+I) Model: PipelineRunner(同名)+ VM: ControlVM / HistoryVM Model 側は MVC 案とほぼ同じ。VM が表示形に整形して View に流す
差分の要点:
  • MVC 案の「Facade としての AppController」は 不要(MainViewModel が代替)
  • 各 Panel(View)から AppController を呼ぶ箇所が消える → Command 経由に統一
  • 状態同期のために手書きしていた連動コード(`SettingsPanel._on_status_from_thread → on_status_change → _apply_status` 等)が Observable の購読に置き換わる
  • 3 秒周期 poll の status refresh は 消える(push 駆動)

7. 段階移行案

8. リスクと留意点

リスク対策
customtkinter の widget が StringVar に対応しない場合の更新コードが増える TkBinder で「subscribe + widget.configure(text=...)」をラップ。コード量は最小化
Observable の listener がメモリリークの温床になる(widget が破棄されても listener が残る) Subscription トークン方式(現状の R2-6 / BackendBase.subscribe を踏襲)。View 破棄時に明示 unsubscribe
auto-persist で保存頻度が上がる DebouncedSaver で 500ms 集約。設定変更 1 回 = ファイル書込み 1 回に近づける
移行中に「MVC 残骸」と「MVVM 新形」が混在 Phase ごとに「MVC 側を完全に消す」プルリク単位を切る。中間状態を長期間放置しない
テスト fixture の書き直しが大きい VM 単体テストは fixture 軽量(Model モックを VM に注入するだけ)。View 統合テストは tk fixture を一本化
「設定の保存ボタン」が消えたことのユーザ混乱 NotificationBanner で「変更は自動保存されます」を初回起動時に出す or LogSettingsView にヒント表示
BackendBase の listener 機構を Observable[T] に extraction する際の既存テスト破壊 Phase 1 の最初のコミットで BackendBase の内部実装だけを Observable[T] 委譲に置き換える(公開 API subscribe() / get_status() は維持)。既存 tests/test_backend_base.py がそのまま pass することを完了条件にする

8-x. 代替案ノート — なぜ traitlets / blinker を使わないか

検討経緯(2026-06-09 ユーザとの対話):
選択肢評価不採用の理由
traitlets(Jupyter 実績、descriptor ベース) 機能的には適合するが… - BackendBase.subscribe 系統(既存)と HasTraits / @observe(traitlets)の 2 流儀が共存することになる
- traitlets 採用で削減できる自前コードは ~30〜50 行(descriptor 糖衣分)。一方 extraction なら ~200 行で全部揃う
- StringVar ブリッジ / AsyncCommand は traitlets でも自前必要 ⇒ 「依存を入れた割に削減量が小さい」
blinker(Signal/Slot) 軽量だが signal 部分しか提供しない。Observable[T] / Command は結局自前で書くことになり、依存を入れる動機が弱い
RxPY(reactivex) 合わない ストリーム指向(event stream)で、値プロパティの変化通知という MVVM ViewModel の用途には形が違う
atom(enaml 基盤) 性能は最高だが enaml 抜きでの単独利用は情報が少なく、Tkinter プロジェクトに持ち込むメリットが薄い
extraction(本計画の採用案) 採用 BackendBase 既存パターン(_listeners_lock + listener コピー後ロック外通知 + Subscription トークン)が そのまま Observable[T] の実装になる。 新規依存なし、流儀が 1 つに揃う、テスト保守コストが既存延長で済む。 自前総量 ~200 行は、traitlets 採用時の自前ぶん(~100〜150 行) + 依存学習コスト + パラダイム共存コストを上回らない
1 行要約: 本プロジェクトは BackendBase という形で 既に Observable パターンを持っている。 これを generic 化すれば「自前実装」ではなく「既存パターンの一般化」になる。外部ライブラリ採用は 「別パラダイムを併存させる」コストの方が高くつく、と判断した。
再検討トリガ: Phase 1 で Observable[T] の自前実装に予想外の難所 (リーク / 競合 / ガベージコレクト周りのエッジケース)が出た場合は、当該 Phase で traitlets 導入を再評価する。 その時点で extraction を捨てる必要はなく、Observable[T] の内部実装だけ traitlets に差し替えれば 公開 API は維持できる(本計画の Observable インタフェース設計はそれを許す前提で組む)。
