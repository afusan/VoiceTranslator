3. 分割提案

現状 9 責務を 1 クラスに詰めているのを、5 つのコラボレータ + Facade としての AppController に再構成する案。 依存方向は「GUI 側 ← Facade ← 内部コラボレータ」を守る。

3-4. 想定リスクと対策

リスク対策
GUI が AppController の現 API に多数依存しているため、I/F を変えると破壊的 AppController を Facade として残し、API を 100% 維持。中身だけ委譲に書き換える
コラボレータ間の循環参照(SettingsFacade が backend 切替で BackendLoaderService.evict を呼びたい等) 「設定変更を通知する Hook」を SettingsFacade から発火し、BackendLoaderService がそれを購読する形(Observer)
テスト fixture が一気に書き直しになる 段階的に進める: まず StatusBroadcaster だけ抽出 → テスト追加 → 次に CredentialsCoordinator 抽出 …
分割しすぎて Facade が薄い委譲だけになる(「Pass-through hell」) 横断的フロー(verify_and_save → 状態更新 など)は Facade に集約。小さい I/F は直接コラボレータを露出する選択肢も検討

4. 段階分割の推奨順

#抽出する責務難易度恩恵
1 StatusBroadcaster(A+C) 小 multi-listener / status_summary がまとまる。test fixture 軽量化の恩恵大
2 CredentialsCoordinator(D) 小 外部依存(CredentialsStore)が閉じる。Phase E-2 の責務を 1 箇所に
3 BackendLoaderService(F + E) 中 1, 2 ができていれば evict 時の状態通知が綺麗に書ける
4 SettingsFacade(B) 中 3 の BackendLoaderService と Observer で繋ぐ
5 PipelineRunner(G+H+I) 中〜大 最後にやる(Coordinator + Logger 群 + ハンドラを一気に動かすので影響範囲が広い)

※ 各段階で AppController の公開 API は維持(GUI 側を触らず済む)。 コミット単位は 1 段階 = 1 ブランチ。
