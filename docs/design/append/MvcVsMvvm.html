MVC と MVVM のアーキテクチャ的な違い

作成: 2026-06-08。Python(特に Tkinter / customtkinter 系)での実装視点で整理。

1. 構造の図解

MVC(Model-View-Controller)

View(画面)
ボタン・テキストボックス・ラベル等のウィジェット
↕
Controller(操作の橋渡し)
View からのイベントを受け、Model 更新と View 再描画を仲介
↕
Model(状態 + ドメインロジック)
設定値、計算結果、外部 API の結果など。View を知らない
  • ① View が「ボタン押された」イベントを Controller に通知
  • ② Controller が Model を更新
  • ③ Controller が **明示的に** View を書き換える(label.config(text=...))
  • View ⇄ Controller は強結合:Controller は View インスタンスを直接参照する
  • 状態の所在:Model に集中。View は描画のたびに引く

MVVM(Model-View-ViewModel)

View(画面)
ViewModel の Observable プロパティに**バインド**(購読)
↕ バインディング
ViewModel(UI 用の状態)
表示用に整形された状態 + Command。View を知らない
↕
Model(ドメイン)
純粋なドメイン。UI のことを一切知らない
  • ① View がボタンを Command にバインド(ViewModel.click_handler 等)
  • ② Command が ViewModel の Observable プロパティを書き換える
  • ③ プロパティ変更が 自動で View に伝播(=データバインディング)
  • View ⇄ ViewModel は疎結合:ViewModel は View インスタンスを知らない
  • 状態の所在:ViewModel に「UI 用の整形済み」が居る。Model は純粋なドメイン
1 行要約: MVC は「Controller が View を更新する 命令型」、MVVM は「View が ViewModel の状態を 購読 して自動更新される 宣言型」。

2. 同じ機能を両方で書くと(擬似コード)

「テキストボックスに名前を入れて『挨拶』を押すと、ラベルに『Hello, <名前>』と表示」を両方の流儀で書いた例。

MVC 流(Tkinter)

# View: ウィジェットの組み立てだけ
class GreetingView:
    def __init__(self, master, controller):
        self.controller = controller
        self.name_entry = tk.Entry(master)
        self.greeting_label = tk.Label(master, text="")
        tk.Button(master, text="挨拶", command=self._on_click).pack()
        # ↑ View が Controller を直接 知っている

    def _on_click(self):
        name = self.name_entry.get()
        self.controller.greet(name)  # ① Controller に通知

    def update_greeting(self, text):
        self.greeting_label.config(text=text)  # ③ Controller から呼ばれて View 更新


class GreetingController:
    def __init__(self, view, model):
        self.view = view   # ← View インスタンスを保持(結合点)
        self.model = model

    def greet(self, name):
        self.model.last_name = name           # ② Model 更新
        self.view.update_greeting(            # ③ View を明示的に更新
            f"Hello, {name}"
        )

MVVM 流(疑似コード、enaml / Qt 風)

# ViewModel: Observable プロパティ + Command
class GreetingViewModel(Observable):
    name = ObservableProperty("")             # 入力
    greeting = ObservableProperty("")         # 出力(自動でラベルへ反映)

    def greet_command(self):                  # Command
        self.greeting = f"Hello, {self.name}"
        # ↑ プロパティを書き換えるだけ。View 側は自動更新

# View(宣言的)
<Entry text="{viewmodel.name}" />            # ← 双方向バインド
<Label text="{viewmodel.greeting}" />        # ← 一方向バインド
<Button text="挨拶" command="{viewmodel.greet_command}" />
# View は ViewModel を「購読」するだけで、ViewModel は View を知らない
差分のポイント:
  • MVC は「Controller が View に label.config(text=...) と書く」必要がある
  • MVVM は「ViewModel が self.greeting = ... を書くだけ」で View が自動追従する
  • MVC の Controller は View インスタンスを参照、MVVM の ViewModel は View を 一切知らない

3. 観点別の対比

観点 MVC MVVM
View ⇄ 中間層 の結合 Controller が View を 直接参照 して更新 ViewModel は View を 知らない(Observable のみ)
状態同期の方式 命令型(Controller がいちいち書く) 宣言型(データバインディング、自動)
状態の所在 Model に集中、View はその都度引く ViewModel に UI 用の整形済み状態が居る(キャッシュ的)
テスト(ロジック) Controller の単体テストは View モックが要る ViewModel は GUI なしで純テスト可能
学習コスト 低(Observer すら無くても動く) 中(バインディングの理解が要る)
必要な道具 標準ライブラリだけで OK バインディングを提供するフレームワーク or 自前 Observer 実装
UI 変更時のコード波及 Controller を毎回さわる View 宣言だけ書き換えれば済むことが多い
大規模化での見通し Controller が肥大化しがち ViewModel 単位で分割しやすい
Python での実装難度 Tkinter で素直に書ける Tkinter には公式バインディングなし、Qt 系が必要

4. 実装上の決定的な違い:「データバインディング」の有無

MVVM の心臓部は データバインディング。これがフレームワーク側で提供されているかで「MVVM が使える/使えない」が決まる。

フレームワークデータバインディングMVVM 採用しやすさ
PyQt6 / PySide6 + QML公式提供(Qt Quick)公式パターン
PyQt6 / PySide6(Widget のみ)Signal/Slot で半自動手で組める
enaml + atom公式提供(宣言的)小規模だがちゃんと動く
FletReactive スタイルMVVM 風に書ける、実績はまだ少
Tkinter / customtkinter無し(StringVar 程度)自作必須
KivyKV 言語で提供採用可
Tkinter の StringVar はバインディングか?
- StringVar + textvariable= は「Entry の値 ↔ 変数」の双方向同期。これは限定的なデータバインディング。
- でも「ViewModel の Observable プロパティが N 個あって全部の View に自動反映」までは行かない。
- 結論:部分的にはバインディング相当 だが、本格 MVVM の自動同期能力には届かない。

5. このプロジェクトでの判断

判断項目結論
customtkinter のままで MVVM 化できるか 自作の Observer を書けば「MVVM 風」までは到達できる。ただし本格的なデータバインディングを欲しがると痛い
Qt 系に移行すべきか 現状の規模では UI 全書き換えのコストに見合わない。移行は非推奨
現状の実装をどう位置付けるか 「MVC + Observer 混在」 が最適。AppController 分割案(5 コラボレータ)はこの方向の延長
必要なら何を入れるか blinker(Signal/Slot 軽量実装)を導入して Observer を明示化する程度で十分
