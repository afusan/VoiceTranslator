複合 backend 候補(複数ロール一括)

backendCandidates.html(レイヤ別の単体 backend 候補)の姉妹版。 ここでは「連続する複数ロールを 1 つの backend が一括処理する」候補を扱う。 編成表機構(covers_roles 申告 → ロール吸収)の上に乗るもので、 緑背景 = 実装済み / 候補 = 依存追加や機構拡張が 必要になった時点で着手 / 対象外 = 現機構と形が合わない。 価格は変動するため目安(発注前に公式確認)。

1. 複合の型と編成機構の対応状況

ロール組合せ機構の対応必要な追加(未対応の場合)
ASR + Translator(音声 → 翻訳テキスト) 対応済み なし(複合 I/F AsrTranslatorBackend + processor 実装済み。新候補は backend 実装と登録のみ)
TTS + Output(テキスト → 直接再生) 機構は素通し可・I/F 未定義 複合 I/F(例: synthesize_play(text, lang))+ _process_tts_output。終端の一般則(Output を含む = on_utterance_done)はそのまま使える
Translator + TTS(原文テキスト → 翻訳音声) 同上 複合 I/F + processor。LLM の 1 ショット(翻訳+音声生成)が現実的になった時点で検討
Capture + VAD (+ ASR)(ストリーミング) スコープ外 request/response 前提の入力ステージ・リトライと形が合わない(再接続は backend 内責務になる)。入力ステージ融合の構造だけ温存済み

3. TTS + Output / Translator + TTS 候補(機構拡張後)

名前組合せ形態認証備考
SAPI 直接再生(pyttsx3 say/runAndWait) TTS + Output ローカル 不要 最小の動作確認用候補。OS 既定デバイスへ直接再生するため出力デバイス選択が効かなくなる制約を UI にどう出すかが論点
ElevenLabs streaming playback TTS + Output クラウド API key stream API + ローカル再生ヘルパ。合成完了を待たずに鳴り始めるためレイテンシ短縮の検証価値あり
GPT audio(modalities=audio) Translator + TTS クラウド API key(OpenAI) テキスト入力 → 翻訳 + 音声生成を 1 コール(出力 audio)。gpt_audio_translate と同じ呼び出し系で実装は軽い。音声出力トークンは入力より高単価

4. 選定(2026-06-10)

  • 実装済み 3 パターンで「複合の動作確認の品揃え」を構成:
    • faster_whisper_translate — ローカル・無認証(縦通しの基準)
    • openai_whisper_api_translate — クラウド・認証あり・英語固定 / 原文なし型
    • gpt_audio_translate — クラウド・認証あり・任意言語 / 原文あり型
  • 選定理由: OpenAI 系 2 種は既存依存(httpx)と既存の認証フロー(単一 API key)を そのまま流用でき、追加インストール無しで「認証あり複合」の実機確認ができる。 また「英語固定×原文なし」「任意言語×原文あり」で複合の表示・言語連動の両極をカバーする。
  • API key の使い回し: 自前 namespace(openai_whisper_api_translate / gpt_audio_translate)を優先し、未設定なら同じ OpenAI key を使う既存 backend (openai_whisper_api / openai_gpt)の保存値に fallback する。
  • 見送り(トリガー待ち): SeamlessM4T(重依存・GPU 前提)/ Azure(専用 SDK + ストリーミング I/F)/ Gemini(google-genai 依存追加)。TTS+Output 系は複合 I/F と processor の追加が前提のため、必要になった時点で機構拡張とセットで着手。
