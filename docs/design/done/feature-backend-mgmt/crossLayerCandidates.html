レイヤを跨ぐ統合 backend(I/O 明示版)

現アーキテクチャは 6 レイヤ分離(Capture / VAD / ASR / Translator / TTS / Output)。 本表は 複数レイヤを 1 つの実体で呑む統合型ライブラリを、入出力を明示して カタログ化したもの。収め方の設計は本表ではしない(Adapter / Megabackend / ルート分岐は別議論)。
観測性◎ = 内部の src_text/tgt_text 等を API で取り出せる / ○ = 一部取れる / × = 取れない(ブラックボックス)。
内部標準フォーマットは 16kHz / mono / float32(`common/types.py`)。サンプルレート/dtype が 異なる backend はリサンプル/フォーマット変換が必要。

既存 backendCandidates.html(単一レイヤに収まる backend のスペック比較)とは 別用途。本表は「跨ぎパターン × I/O」の整理で、収め方検討の素材。

横断観察(設計議論の素材)

  • 「発話単位」vs「連続ストリーム」の二分法: 現パイプラインは発話単位(seq_id ベース)。VAD+ASR 統合のクラウド系と全部入り系は 本質的にストリーム。Section 1 と Section 5 は時間モデルが根本的に異なる。
  • 観測性は「クラウドが意外と◎」: Azure Speech Translator は同時通訳用途のせいで中間段イベントを明示的に出す。 一方ローカルの SeamlessM4T S2ST は「モデル別モードを 2 回呼べば取れる」(計算コスト 2 倍)。 OpenAI Realtime は input_audio_transcription オプションで src_text を取れる。
  • 入力フォーマット差: 現 MVP は 16kHz / mono / float32。Realtime API は 24kHz / pcm16、Vosk は 16kHz int16 等、 バラバラ。Capture レイヤ直後にフォーマット変換アダプタが必要になりがち。
  • 言語制約の差: Whisper translate は 英語固定。SeamlessM4T S2ST は出力 36 言語のみ (日本語 OK)。LLM 系(GPT-4o / Gemini)はプロンプトで任意言語。
  • 「単一ライブラリで Translator+TTS だけ」は希少: Section 3 が薄いのはこのため。クラウドはサービス内に組み込まれている。
  • VAD 抜き / Capture 抜き: Capture(デバイスからの取得)を呑む統合 backend は事実上ない (音声デバイス I/F は OS 依存)。Capture は分離維持で OK。

次のステップ: この表を見て、現アーキテクチャ(レイヤ分離)にどう収めるかを議論する。 選択肢は (a) Adapter で複数レイヤに同一実体登録、(b) Megabackend で新レイヤ概念、 (c) Coordinator にルート分岐を持たせる、など。本表は判断材料の提供までで、設計は別途。
