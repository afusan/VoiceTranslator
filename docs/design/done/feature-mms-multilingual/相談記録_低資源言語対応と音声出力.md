## A. 低資源言語対応

### 現状の棚卸し(レイヤ別のギャップ)

| レイヤ | 現状 | アフリカ系言語のカバー |
|---|---|---|
| 翻訳 | **NLLB-200(既定)** | ◎ **実はすでに最強**。NLLB は Meta が低資源言語のために作ったモデルで、200 言語中アフリカ系を約 55 言語含む(スワヒリ語・ヨルバ語・ハウサ語・アムハラ語・ズールー語・ウォロフ語 等) |
| ASR | faster-whisper | △ Whisper は約 99 言語。スワヒリ語等は対応するが低資源言語ほど精度が落ち、未対応言語も多い |
| TTS | SAPI / Piper / クラウド 3 種 | ✕ ここが最大のギャップ。SAPI は OS 搭載 voice 依存、Piper は約 30〜40 言語(スワヒリ語はあるがヨルバ語・ハウサ語等は無い)、クラウド系も主要言語中心 + 従量課金 |
| UI | 出力言語の候補 | ✕ `_TGT_LANG_CHOICES` が **16 言語のハードコード**。NLLB は 200 言語いけるのに UI が絞っている |

### A-1. TTS は言語別にライブラリが必要か → **不要。「ライブラリ別」ではなく「モデル(voice)別」**

現代のニューラル TTS は「エンジン 1 つ + 言語別の voice モデル」という構造です。
すでに入っている Piper がまさにこの形(ONNX エンジン 1 つ、voice は言語ごとに初回 DL)。
つまり**アーキテクチャ上の追加負担はゼロ**で、問題は「低資源言語の voice モデルが
存在するエンジンを選べているか」だけです。本アプリの Strategy 構成
(`supported_languages` 申告 → 出力言語プルダウンに AND 連動)もそのまま機能します。

### A-2. 包括的解決案 → **ある。「Meta MMS-TTS backend を 1 つ足す + UI の言語表を開放」**

- **Meta MMS(Massively Multilingual Speech)**: TTS が **1,100+ 言語**(アフリカ系を大量に
  含む)。HuggingFace transformers の VITS モデルで、**言語ごとに ~100〜150MB 級、
  CPU で動く**(非自己回帰なので CPU でも速い)。「エンジン 1 つ + 言語別小型モデル
  (初回 DL)」なので Piper と同じ流儀で backend 1 クラスとして実装できる。配布方針
  (CPU floor / ローカル動作 / 従量課金なし)とも完全に整合。
- これと **NLLB の 200 言語を UI に開放**(言語表の拡張。翻訳側はコード変更ほぼ不要)を
  組み合わせると、「翻訳 200 言語 × TTS 1,100 言語」になり、貧困国ユースの
  「オフライン・無料・低スペック PC」という制約にもそのまま乗ります。
- 将来 ASR がボトルネックになったら **MMS-ASR**(同じく 1,100+ 言語)を ASR backend と
  して追加する道もある(同一プロジェクトなので流儀も揃う)。
- 究極のフォールバックとして **eSpeak-NG**(フォルマント合成。ロボ声だが 100+ 言語・
  数 MB・完全オフライン)という選択肢もある。「聞き取れれば良い」用途の保険。

### 注意点(実装前に要確認)

1. **MMS のライセンスは CC-BY-NC 4.0(非商用)**。アプリ本体(MIT 等)とは別に
   「このモデルを使う場合は非商用」という条件がユーザに付く。GitHub 公開の無償アプリ
   としては概ね問題ないが、README / 同意ダイアログでの明示が必要(クラウド同意
   ダイアログの既存機構が流用できる)。
2. 言語コードの正規化: NLLB は `swh_Latn` のような FLORES コード、Whisper/MMS は
   ISO 639 系。共通言語テーブル(`common/languages.py`)の拡張が本丸になる。
3. 音質は言語により差がある(低資源言語ほど「明瞭だが棒読み」寄り)。

**規模感**: MMS-TTS backend 追加 = 1〜2 日級(Piper backend が良いテンプレート)。
UI 言語表の開放 = 半日〜1 日級(16 → 200 にするとプルダウン UX の工夫も要る:
検索可能リスト or よく使う言語 + その他)。

## B. 翻訳音声を Teams / Discord / Slack へ出力する一般的な方法

### 方式比較

| 方式 | 仕組み | 対応アプリ | コスト |
|---|---|---|---|
| **1. 仮想マイク(本命)** | 仮想オーディオデバイス(VB-Audio Virtual Cable 等)に TTS を出力し、相手アプリのマイク入力に「CABLE Output」を選ばせる | **全アプリ共通**(Teams/Discord/Slack/Zoom/ゲーム何でも) | ほぼゼロ(後述) |
| 2. アプリ別 bot/API | Discord: bot がボイスチャンネルに参加して音声再生(公式 API あり)。Teams: Graph 通信 API(Azure bot + 組織管理者の同意が必要で重い)。Slack: ハドルへの音声注入 API は**存在しない** | アプリごとにバラバラ | 高(アプリ別実装 + 認証) |
| 3. テキストで出す | 翻訳テキストを Webhook で chat に投稿(Slack/Discord/Teams とも Webhook は簡単) | 全アプリ | 低(ただし「音声」ではない) |

### 推奨: 方式 1(仮想マイク)

これは Voicemod 等のボイスチェンジャーが使っている業界標準の方法で、しかも
**本アプリは今の機能でほぼそのまま実現できます**:

```
[実マイク] → 入力デバイス(本アプリ) → VAD → ASR → 翻訳 → TTS
                                          → 出力デバイス = 「CABLE Input」
[Teams/Discord] のマイク設定 = 「CABLE Output」
```

- 入力デバイス = 実マイク、出力デバイス = 仮想ケーブル、を選ぶだけ。アプリ側の
  コード変更は不要で、**まず VB-Cable を入れて手元で試せる**(導入手順を manual に
  足すだけで「対応」と言える)。
- 逆方向(相手の音声を翻訳して自分が聞く)は、既にある ProcTap(プロセス単位
  キャプチャ)がそのまま該当する。つまり双方向とも既存設計の延長線上。
- 改善するなら小粒の追加機能で済む:
  - **モニタ出力**(仮想ケーブルへ送りつつ自分のヘッドホンでも翻訳音声を確認する
    二重出力)— Output レイヤの小拡張
  - 「仮想マイクセットアップ」の手順ガイド(manual / 初回ダイアログ)
- 注意点:
  - VB-Cable はドネーションウェアで**同梱再配布にはライセンスが要る**ため、
    「ユーザが別途インストールする」案内方式が無難(独自仮想オーディオドライバの
    自作は署名付きカーネルドライバが必要で非現実的)。
  - 会話用途ではレイテンシ(発話終了 → 翻訳音声)が体感を支配する。現行の計測機構
    (processtime.csv)がそのまま効く。

### 本体機能か別枠か

パイプライン(マイク → … → TTS → 出力デバイス)は**既存パイプラインそのもの**なので、
別アプリにする理由は薄いです。「出力先が仮想ケーブルである」のは設定の一形態に
過ぎず、本体機能(+manual の構成例 1 ページ)が自然です。方式 3(テキスト Webhook
投稿)だけは独立性が高いので、必要になったら Output 系の新 backend(`webhook` 等)
として足せます。

### まず試すなら(実装ゼロ)

1. VB-Cable をインストール
2. 本アプリ: 入力 = 実マイク / 出力 = `CABLE Input` / 出力言語 = 相手の言語
3. Discord 等: マイク = `CABLE Output`
これで動くはずです。動けば「対応」はドキュメント整備だけになります。
