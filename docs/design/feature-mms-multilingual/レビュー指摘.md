ASR/Translator/TTS 各 backend の境界変換も一貫している。**致命的バグ・クラッシュは未検出**。
branch 関連 small テスト 233 件 green を再現確認。

以下は「仕様としては妥当だが期待とズレる/堅牢性が薄い」系の指摘。**High はなし**、Medium が

## Medium

### M-1. MMS は日本語を読み上げられない(本アプリの主用途が成立しない)
MMS∩NLLB の 99 言語に **`jpn` / `zho` / `ara` / `ita` が含まれない**(検証済み。MMS の
チェックポイントに存在しないため)。本アプリは CLAUDE.md / fallback 規則(`jpn > eng`)が
示すとおり**日本語出力が主用途**だが、TTS に `mms` を選ぶと:
- 出力言語候補(`restrict_to_tts(NLLB, MMS)`)から `jpn` が消える。
- 現在値 `jpn` は `compute_tgt_selection` で `eng` へ自動 fallback(バナーは出る)。

つまり「日本語で喋らせたいユーザが MMS を選ぶと、黙って英語出力に化ける」。fallback 機構
自体は機能するが、**`docs/manual.md §5-4b` はこの制限(主要言語が抜ける)を書いていない**。
スワヒリ等を読み上げる用途には正しい backend だが、「日本語/中国語/アラビア語は MMS 非対応 →
これらの出力には SAPI/Piper/クラウドを使う」を manual に明示すべき。
→ 仕様は妥当。**ドキュメントの期待ギャップ**として要追記。

### M-2. MMS の device 解決が MPS(Apple Silicon)を無視
`MmsTtsBackend._resolve_device()`(`tts/mms_backend.py:120-129`)は `cpu`/`cuda` しか返さず、
`torch.cuda.is_available()` が偽なら無条件 `cpu`。共有ヘルパ `common/device.py:resolve_torch_device`
は **mps を auto 解決する**のに、MMS は同じ torch 系なのに自前実装で MPS を落としている。
- 配布方針(CLAUDE.md「配布方針」: GPU/アクセラレータがあれば自動使用=**Apple Silicon MPS 含む**、
  「device 切替はコードパス1本」)に反する。
- `device="mps"` を**明示**しても `_resolve_device` の分岐(`in ("cpu","cuda")`)を外れ、cpu に落ちる。
→ NLLB と同じく `resolve_torch_device` を使えば一掃できる。VITS×MPS の実機懸念があるなら、
   その判断(なぜ自前で MPS を外したか)を docstring に残すべき(現状は理由の記載なし)。

### M-3. 言語パックのロード失敗が `FatalError` でパイプライン全停止
`_load_voice`(`tts/mms_backend.py:171-204`)は DL/初期化失敗時に **`FatalError`** を投げる。
MMS の言語ロードは**会話中(初回発話)に起こり得る**(prefetch されなかった/別言語へ切替直後)。
- 一過性のネットワーク断で 150MB の DL が落ちると、`FatalError` → パイプライン即停止。
- 起動時 1 回ロードの NLLB が Fatal なのは妥当だが、MMS は**実行中の動的ロード**なので、
  `RecoverableError`(リトライ)か `SkipError`(当該発話だけ捨てる)の方が縮退として自然。
→ 「prefetch が裏で確保するから実害は小さい」という前提だが、prefetch 失敗時の本線が Fatal。

### M-4. 言語ロード失敗で backend 全体ステータスを `NOT_DOWNLOADED` に汚す
`_load_voice` の失敗ハンドラは `self._set_status(ModelStatus.NOT_DOWNLOADED)` を呼ぶ
(`tts/mms_backend.py:190, 203`)。MMS はエンジン準備済み(LOADED)で**個別言語**を足す設計
なのに、1 言語のロード失敗で backend 全体が「未 DL」表示になる。
- キャッシュ済みで動作可能な言語があっても UI 上は壊れて見える。成功ロードまで回復しない。
- **背景 prefetch の失敗**(AppController は例外を握る)でも、backend 内部でステータスが先に
  `NOT_DOWNLOADED` に倒れた状態が残る。→ ユーザ操作と無関係にステータスが汚れる。
→ 言語単位の失敗はレイヤ全体状態に波及させない設計が望ましい(集約テキストへ言語名併記で代替、
   は Plan の当初方針とも整合)。

## Low

### L-1. ElevenLabs の Tagalog 申告コード不整合(`fil` vs `tgl`)
ElevenLabs は Tagalog を `"fil"` で申告(`iso1_to_iso3("fil")` は未知 passthrough → `"fil"`)。
NLLB/MMS/Whisper は同言語を `"tgl"` で申告。AND(`restrict_to_tts`)で一致せず、
**ElevenLabs+NLLB のとき Tagalog が出力候補から落ちる**。ElevenLabs は voice 主体で実合成は
コードに依存しないため実害は「候補欠落」のみ。**移行前(`fil` vs `tl`)からの持ち越し**だが、
今回の正準化で `tgl` に揃った他 backend との差として残った。`ISO1_TO_ISO3` に `"fil"`→`"tgl"`
等の別名吸収を入れるか、ElevenLabs 側を `"tgl"` 申告にすれば解消。

### L-2. `set_setting` は言語コードを正準化しない
正準化(`_normalize_language_codes`)は **`ConfigStore.load()` 時のみ**。`set_setting("languages",
"tgt", X)` は X をそのまま保存する。GUI 経路(プルダウン/🔍)は常に正準 639-3 を渡すので
実害はないが、プログラム経由や旧値の直接書込みで 639-1("ja")が入ると、次回 load まで
正準化されず、その間 **MMS の prefetch は no-op / synthesize は SkipError** になる。
防御の薄さ(正準化を書込み境界にも置くか、最低限テストで「set→get が正準」を固定)。

### L-3. テストが 639-1 コードで書かれている(移行後の実コードと乖離)
`tests/test_mms_backend.py` の LRU/キャッシュ系が `"en"/"fr"/"de"`(639-1)で検証。
`_ensure_language` はメンバーシップ非検証(`_load_voice` がゲート)のためモック差替えで通るが、
**本番が渡すのは 639-3**(`"eng"` 等)。テストが移行後の実コードパスを表していない。
`test_returns_sorted_iso639_1_subset` の名称・docstring も 639-1 のまま stale。

### L-4. large テストの samplerate ハードコード
`test_mms_tts_large.py` が `sr == 16000` を固定アサート。MMS は多くが 16kHz だが普遍ではない。
eng/swh では正しいが、対象言語を増やすと脆い。

### L-5. クラウド backend の既定言語が 639-1 表記のまま
`backend_setup.py`(google_stt=`"en"` L425 / google_tts=`"en"` L664)・
`layer_settings_schema.py`(L432, L585=`"en"`)。各 backend の `__init__` が
`iso1_to_iso3` で正準化するため動作はするが、**内部標準 639-3 と既定値の表記が不一致**。
schema 既定は `"eng"` が一貫(GUI 編集時に 639-1 を見せない)。

### L-6. 同一言語の同時ロードで重複 DL/構築
`_ensure_language` はロック外でロード(意図的=重い DL 中に他スレッドを止めない)。
同一言語を prefetch と synthesize が同時要求すると **150MB の DL/構築が二重**に走り、
後勝ちで一方を破棄。コメントで認知済み・正しさは保たれるが、稀に無駄が出る。

## 情報(対応不要・把握のみ)

- **I-1**: Whisper 対応だが NLLB 非対応の 14 言語(`afr, bre, haw, lat, lin, ltz, mlg, mri, nob,
  oci, san, snd, tuk, yid`)を src に選ぶと、NLLB の src コードが `eng_Latn` に fallback し
  **源言語を誤認した翻訳**になる。既存挙動(NLLB が元から非対応)で 639-3 化による回帰ではない。
  `ISO_TO_NLLB` 拡充 or src 候補を「翻訳もできる言語」に絞る将来課題。

  スクリプト(実行時非依存)だが、transformers 版更新で壊れ得る。再生成時に要確認。
- **I-3**: `backends_config` に `mms` の既定エントリなし(piper/elevenlabs と同様 `_read_str`
  の default で吸収)。`auto_load` 既定が無く常に手動ロード=既存 backend と整合。Plan Phase4 の
  「既定値補完」チェックは `_read_str` 依存で済ませた、で整合と判断。
