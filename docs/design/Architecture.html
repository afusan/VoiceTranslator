1-2. スレッド構成(B+案)

UI フリーズ防止と取得ラグ抑制のため、以下の 5 スレッド構成で動かす。

UI (main)
tkinter mainloop / 設定操作
↓ start ボタン押下
Loader (1回限り)
各バックエンドのモデルロード
↓ ロード完了で3スレッド起動
Input thread
capture → VAD → q1
q1 (max 3)
あふれ=最古を捨てる
Process thread
ASR → 翻訳 → TTS → q2
q2 (max 3)
あふれ=最古を捨てる
Output thread
再生 + UI通知
スレッド役割備考
UI (main)tkinter mainloop。ユーザ操作と画面更新のみ。常に応答可能を維持
Loader各レイヤのバックエンドを順にインスタンス化(モデルロード)。完了/失敗を UI に通知。1回限り起動・終了
Inputcapture.read_chunk → vad.process で Utterance を作って q1 に投入。取りこぼし最小化
Processq1 から取り出し、ASR → 翻訳 → TTS を直列実行して q2 に投入。処理が重い段
Outputq2 から取り出して再生、on_utterance_done コールバックを発火。処理中の発話と並行再生可

キューあふれ時は最古の発話を捨てて新しい発話を優先する(リアルタイム性確保)。 発話の取りこぼしが続く場合は Process 側のレイテンシ過大のシグナルとしてログに記録する。

停止シーケンス

  1. stop_event をセット → Input スレッドが次の read_chunk 完了後に終了
  2. q1 にセンチネルを投入 → Process スレッドが終了
  3. q2 にセンチネルを投入 → Output スレッドが終了
  4. 各スレッドを join → capture.stop() / output.stop() を呼んでデバイス解放

モデル状態(UI公開)

各レイヤのバックエンドについて以下の 3 状態を AppController が保持し、GUI に色付きラベルで表示する(英語固定)。

状態表示意味
NOT_DOWNLOADED"Not Downloaded"(赤)ローカルキャッシュ無し。起動時にDLが走る。
LOADING"Loading..."(オレンジ)DL or メモリへロード中。
LOADED"Loaded"(緑)メモリにロード済み or キャッシュ済み(即ロード可)。

2. 各レイヤの役割

レイヤ 役割
AudioCapture 選択された入力デバイスから音声を取得し、内部標準フォーマット(16kHz/mono/float32)に正規化してパイプラインに供給する。
VAD 連続するPCMストリームから発話区間を検出し、無音で区切って「1発話 = 1 Utterance」として下流に流す。
ASR 発話単位の音声を入力言語のテキストに書き起こす。出力先言語の指定はしない(=翻訳しない)。
Translator 書き起こしテキストを src言語 → tgt言語 に翻訳する。
TTS 翻訳テキストを音声(PCM)に合成する。
AudioOutput 合成音声を、入力とは別の出力デバイスで再生する。

横断機能

機能役割
AppControllerGUI と内部モジュールの仲介。Loader スレッド起動 / バックエンド生成 / モデル状態の発信 / 設定操作の窓口。
PipelineCoordinatorInput/Process/Output の 3 スレッドを起動・停止し、上限付きキューで連携。
ConfigStore選択中のバックエンド、デバイス、言語ペア、ログ出力先などの設定を YAML で永続化。
Loggerアプリ全般のログ + 翻訳履歴の jsonl 出力。出力先と ON/OFF は設定から取得。
ErrorHandler例外を 致命 / 回復可能 / スキップ可能 / 警告 の4分類で振り分ける。詳細は ErrorCatalog.md(未作成) 参照予定。
cache_check各バックエンドのモデルがローカルキャッシュにあるかを軽量判定。GUI の初期ステータス表示に使う。
GUI設定変更・動作開始停止・直近状態 + モデルステータスの可視化を提供する(customtkinter)。

3. レイヤ間でやり取りする情報 (I/F)

レイヤ間は Utterance という発話単位データを介して受け渡す。各ステージで該当フィールドを追記していく形にし、計測用に timeline に各時点のタイムスタンプを打つ。

境界 流れるデータ 形式 / 仕様
AudioCapture → VAD PCMチャンクの連続ストリーム 16kHz / mono / float32 [-1.0, 1.0]、固定サイズチャンク(例: 20〜30ms)
VAD → ASR 発話単位PCM (Utterance に格納) Utterance.pcm(可変長), timeline: t_capture, t_vad_end
ASR → Translator 書き起こしテキスト + 言語情報 Utterance.src_text: str, src_lang: str, timeline.t_asr
Translator → TTS 翻訳済みテキスト + 言語情報 Utterance.tgt_text: str, tgt_lang: str, timeline.t_translate
TTS → AudioOutput 合成済み PCM Utterance.tts_pcm(エンジン依存レート/形式), timeline.t_tts
AudioOutput → デバイス 再生されたチャンク 出力デバイスのネイティブ形式(必要なら出力レイヤで変換)、timeline.t_playback
(終端) → Logger 翻訳1件のログ jsonl 1行: {ts, src_text, tgt_text, src_lang, tgt_lang, latency_ms, ...}
Utterance(発話単位データ)について
各ステージで追記される共通データクラス。タイムスタンプは timeline.mark(stage_name) で monotonic() を打つ。
詳細フィールド定義は Class.md を参照。

4. データフロー(発話1件あたり)

  1. AudioCapture: PCMチャンクを連続取得し、内部標準形式に正規化。
  2. VAD: 無音で区切り、Utterance として確定(t_capture, t_vad_end)。
  3. ASR: 書き起こして src_text を付与(t_asr)。
  4. Translator: 翻訳して tgt_text を付与(t_translate)。
  5. TTS: 合成PCMを生成(t_tts)。
  6. AudioOutput: 出力デバイスで再生(t_playback)。
  7. Logger: jsonl に1行記録(レイテンシ算出)。

5. 拡張ポイント

  • 新しいバックエンドを追加するときは、対応する抽象I/Fを実装し BackendRegistry に登録するだけでGUIに出現する。
  • OS別の音声取得は AudioCapture の差し替えで対応(Windows = soundcard、Linux = GStreamer 等)。
  • per-app 音声取得は OS別バックエンドの一部として将来追加(pendList参照)。
  • ローカルLLM翻訳は Translator の追加実装で対応(例: Ollama)。

6. 横断方針

  • 役割の表明: 各クラスのコメント/Docstring 冒頭に1〜2行で役割を明記する(CLAUDE.md 準拠)。
  • エラー: 各バックエンドは内部例外を AppError(severity 付き) に包んで送出。中央で ErrorHandler が4分類に振り分ける。
  • 計測: 全ステージで Utterance.timeline.mark() を呼ぶ。
  • デバイス分離: 起動時に入力デバイス ≠ 出力デバイスを保証(同一なら起動拒否)。AEC は将来オプション。
