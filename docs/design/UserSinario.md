# UserSinario (ユーザシナリオ)

本アプリの想定ユースケースとユーザ操作フローを記述する。

---

## 想定ユーザ

- ローカルPC(MVP は Windows 11)で、リアルタイムに音声を翻訳して聞きたい個人ユーザ。
- 例:
  - 英語のYouTube動画やTwitch配信を、日本語音声で聞きたい。
  - 英語の通話/会議の発言を、自分のスピーカ側で日本語にして確認したい。
  - 自分が日本語で話した内容を、相手に英語音声で届けたい(将来要件)。

---

## ユースケース

### UC-1: PC音声(配信/動画/通話)を翻訳して聞く

**前提**
- 視聴したいアプリ(YouTube/Twitch/Zoom 等)で英語の音声が再生されている。
- アプリの入力デバイスと出力デバイスは別物が用意されている(例: 入力=Speakers、出力=Headphones)。

**フロー**
1. ユーザがアプリを起動する。
2. `SettingsPanel` で以下を選択:
   - 入力デバイス: 配信音声が流れているスピーカ(ループバック)
   - 出力デバイス: 翻訳音声を聞きたいデバイス(別のヘッドホン等)
   - src 言語: English (or auto)、tgt 言語: Japanese
3. `DeviceValidator` が「入力≠出力」を確認(同一なら警告で起動拒否)。
4. ユーザが「開始」ボタンを押す。
5. パイプラインが動き出し、発話単位で書き起こし → 翻訳 → 音声再生が連続する。
6. `ControlPanel` には直近の翻訳テキストとレイテンシが表示される。
7. ユーザが「停止」ボタンを押すと、パイプラインを停止する。

**正常系の結果**
- 入力音声に対して、おおむね数秒以内に翻訳済み音声が出力デバイスで再生される。
- 翻訳前後のテキストが画面に表示され、設定によっては jsonl にも保存される。

---

### UC-2: マイクから話した内容を翻訳して読み上げる

**前提**
- マイクが接続されている。
- 出力デバイスはマイクと同じ系統ではない(フィードバック防止)。

**フロー**
1. ユーザがアプリを起動する。
2. `SettingsPanel` で以下を選択:
   - 入力デバイス: マイク
   - 出力デバイス: スピーカ or 仮想ケーブル(会議アプリへ流すなら)
   - src 言語: Japanese、tgt 言語: English
3. `DeviceValidator` チェック。
4. 「開始」ボタンを押す。
5. ユーザが話す → VADで発話区切り → ASR → 翻訳 → TTS → 出力。
6. 「停止」ボタンで終了。

**正常系の結果**
- ユーザの発話が翻訳されて出力デバイスで読み上げられる。

---

### UC-3: 設定を保存・読み込む

**フロー**
1. ユーザが `SettingsPanel` で任意の設定をした後、「保存」ボタンを押す。
2. `ConfigStore` が YAML ファイルとして設定を書き出す。
3. 次回起動時、自動的に最後の設定が読み込まれる。
4. 別プロファイルを使いたい場合は「読込」から YAML を指定できる。

---

### UC-4: バックエンド(ライブラリ)を切り替える

**フロー**
1. `SettingsPanel` のレイヤ別プルダウン(入力/ASR/翻訳/TTS)から、別のバックエンドを選択する。
2. パイプライン停止状態であれば即座に反映、稼働中なら次回開始時に反映。
3. レイテンシ表示で旧バックエンドとの比較ができる。

※ MVP段階では各レイヤ1実装のみ提供。プルダウンは将来のための器として用意しておく。

## エラー時シナリオ

### EC-1: 入力デバイスと出力デバイスが同一
- `DeviceValidator` が検出 → ダイアログ表示「入力と出力に同じデバイスは使用できません」→ 起動拒否。

### EC-2: ASR/翻訳/TTS の一時的失敗
- `ErrorHandler` が SKIP/RECOVERABLE と判定 → 当該発話を破棄してログ出力 → パイプライン継続。

### EC-3: モデルロード失敗・デバイス消失等の致命的エラー
- `ErrorHandler` が FATAL と判定 → ダイアログ表示 → パイプライン停止。

### EC-4: レイテンシ閾値超過
- `Logger` が WARN として通知バナーを出す。動作は継続。
