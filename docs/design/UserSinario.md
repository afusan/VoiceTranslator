# UserSinario (ユーザシナリオ)

本アプリの想定ユースケースとユーザ操作フローを記述する。

---

## 想定ユーザ

- ローカルPC(MVP は Windows 11)で、リアルタイムに音声を翻訳して聞きたい個人ユーザ。
- 例:
  - 英語のYouTube動画やTwitch配信を、日本語音声で聞きたい。
  - 英語の通話/会議の発言を、自分のスピーカ側で日本語にして確認したい。
  - 自分が日本語で話した内容を、相手に英語音声で届けたい(将来要件)。

---

## ユースケース

### UC-1: PC音声(配信/動画/通話)を翻訳して聞く

**前提**
- 視聴したいアプリ(YouTube/Twitch/Zoom 等)で英語の音声が再生されている。
- アプリの入力デバイスと出力デバイスは別物が用意されている(例: 入力=Speakers、出力=Headphones)。

**フロー**
1. ユーザがアプリを起動する。
2. `SettingsPanel` で以下を選択:
   - 入力デバイス: 配信音声が流れているスピーカ(ループバック)
   - 出力デバイス: 翻訳音声を聞きたいデバイス(別のヘッドホン等)
   - src 言語: English (or auto)、tgt 言語: Japanese
3. `DeviceValidator` が「入力≠出力」を確認(同一なら警告で起動拒否)。
4. ユーザが「開始」ボタンを押す。
5. パイプラインが動き出し、発話単位で書き起こし → 翻訳 → 音声再生が連続する。
6. `ControlPanel` には直近の翻訳テキストとレイテンシが表示される。
7. ユーザが「停止」ボタンを押すと、パイプラインを停止する。

**正常系の結果**
- 入力音声に対して、おおむね数秒以内に翻訳済み音声が出力デバイスで再生される。
- 翻訳前後のテキストが画面に表示され、設定によっては jsonl にも保存される。

---

### UC-2: マイクから話した内容を翻訳して読み上げる

**前提**
- マイクが接続されている。
- 出力デバイスはマイクと同じ系統ではない(フィードバック防止)。

**フロー**
1. ユーザがアプリを起動する。
2. `SettingsPanel` で以下を選択:
   - 入力デバイス: マイク
   - 出力デバイス: スピーカ or 仮想ケーブル(会議アプリへ流すなら)
   - src 言語: Japanese、tgt 言語: English
3. `DeviceValidator` チェック。
4. 「開始」ボタンを押す。
5. ユーザが話す → VADで発話区切り → ASR → 翻訳 → TTS → 出力。
6. 「停止」ボタンで終了。

**正常系の結果**
- ユーザの発話が翻訳されて出力デバイスで読み上げられる。

---

### UC-3: 設定を保存・読み込む

**フロー**
1. ユーザが `SettingsPanel` で任意の設定をした後、「保存」ボタンを押す。
2. `ConfigStore` が YAML ファイルとして設定を書き出す。
3. 次回起動時、自動的に最後の設定が読み込まれる。
4. 別プロファイルを使いたい場合は「読込」から YAML を指定できる。

---

### UC-4: バックエンド(ライブラリ)を切り替える

**フロー**
1. `SettingsPanel` のレイヤ別プルダウン(入力/ASR/翻訳/TTS)から、別のバックエンドを選択する。
2. パイプライン停止状態であれば即座に反映、稼働中なら次回開始時に反映。
3. レイテンシ表示で旧バックエンドとの比較ができる。

※ MVP段階以降、各レイヤに複数バックエンドが登録済み。詳細は `manual.md` 「5-4. レイヤ別の実装選択」を参照。

---

### UC-5: 動作中に翻訳言語を切り替える

**前提**
- パイプラインが動作中(▶ 開始 中)。

**フロー**
1. 動作中のまま、`SettingsPanel` 「翻訳」セクションで入力言語(src)または出力言語(tgt)を変更する。
2. AppController が動作中の Coordinator に新言語を即時転送する。
3. **次の発話から** 新言語が適用される。すでに ASR/翻訳キューに積まれている発話は、古い言語の
   ままで完走する(切替直後に旧言語の翻訳が 1〜2 件出る可能性がある)。

**正常系の結果**
- パイプラインを停止せずに、希望のタイミングで翻訳言語を切り替えられる。

※ バックエンドの切替は本シナリオの対象外(UC-4)。

---

### UC-6: 特定アプリだけの音声を翻訳する(per-process キャプチャ)

**想定シーン**
- BGM や別アプリの効果音が混ざる環境で、**配信アプリだけ** / **Discord の通話だけ** を翻訳したい。
- スピーカループバックだと「全アプリの音がまとまって入る」ので分離不能。

**前提**
- Windows(WASAPI Process Loopback を使うため)。
- 起動時に `--extra capture-proctap` を有効にした uv 環境(`py -m uv sync --extra cpu --extra capture-proctap`)。

**フロー**
1. アプリを起動し、`SettingsPanel` 「バックエンド」セクションで **音声取得 を「プロセス (proctap)」** に切替。
2. 「デバイス」セクションの **「プロセス選択…」ボタン** を押下 → ダイアログが開く。
3. ダイアログには現在音声出力中(`AudioSessionState.Active`)のプロセスのみが列挙される。
4. 対象プロセスを選択 → 「▶ 試聴開始」でレベルメータが動き、本当に鳴っているかを確認できる。
   試聴は本番パイプラインと完全独立(WASAPI ストリームを開かない、pycaw メータのみ)。
5. 「OK」で確定 → ボタンラベルが `PID 1234 ▼` 等に変わる。
6. 「▶ 開始」で本番パイプラインを起動 → 当該プロセスの音だけが翻訳・読み上げされる。

**設計上のポイント**
- PID は **永続化しない**(プロセス再起動で別 PID になるため)。アプリ再起動時は「プロセス未選択」
  状態に戻り、Start ボタンが disable される。毎回プロセス選択ダイアログから選び直す UX。
- 試聴 peak の取得は永続 COM ワーカスレッド(`_PeakWorker`)が 5fps で内部 poll し、
  GUI スレッドは atomic 値を読むだけ。

**正常系の結果**
- 選択したプロセス由来の音だけが翻訳パイプラインを流れ、出力デバイスで読み上げられる。
- 他アプリの BGM や効果音は翻訳されない。

### UC-7: テキスト字幕モード(TTS=(なし))

**想定シーン**
- 視聴中の音声を邪魔したくない / TTS の準備をしたくない / クラウド TTS 課金を避けたい。

**フロー**
1. アプリを起動し、`SettingsPanel` 「バックエンド」セクションで **TTS プルダウンの「(なし)」** を選択。
2. Output 行が自動的にグレーアウトされる(復帰用に選択値は維持)。
3. 「設定を保存」して Stop → Start(または ▶ 開始)で text_only モードが反映される。
4. 動作中は、翻訳結果が `ControlPanel` の履歴に表示される(音は鳴らない)。
5. 音声を出したい構成に戻したい場合は、TTS プルダウンを実 backend(SAPI 等)に戻す。

**設計上のポイント**
- text_only モードでは Coordinator は Input / ASR / Translator の 3 スレッドのみを起動し、
  TTS / Output は起動しない。
- Translator 完了直後に `on_text_ready` を発火し、レジャを即 pop してバッファを解放する。
- TTS / Output レイヤの認証情報チェックも skip されるため、クラウド TTS の認証なしで
  Start できる。

**正常系の結果**
- 入力音声に対して翻訳テキストが履歴ボックスに出続け、音は鳴らない。

## エラー時シナリオ

### EC-1: 入力デバイスと出力デバイスが同一
- `DeviceValidator` が検出 → ダイアログ表示「入力と出力に同じデバイスは使用できません」→ 起動拒否。

### EC-2: ASR/翻訳/TTS の一時的失敗
- `ErrorHandler` が SKIP/RECOVERABLE と判定 → 当該発話を破棄してログ出力 → パイプライン継続。

### EC-3: モデルロード失敗・デバイス消失等の致命的エラー
- `ErrorHandler` が FATAL と判定 → ダイアログ表示 → パイプライン停止。

### EC-4: レイテンシ閾値超過
- `Logger` が WARN として通知バナーを出す。動作は継続。
