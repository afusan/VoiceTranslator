エージェンティック開発ワークフロー

本プロジェクトのチャットラリー方式からエージェンティック方式への移行で合意した 運用フロー全体像を、関係する全エージェントが起動時に参照するために置く。

  • 位置づけ: 固定的な運用設計（可変な教訓＝メモリではない）。よって claudeKnowledge/ ではなく docs/design/ 配下に置く。
  • 読み手: 主エージェント（Claude Code 本体）と、すべての配下サブエージェント （designReviewer など）。各 agent.md の「起動時にすること」で本ファイルの読み込みを義務づける。
  • 背景・議論ログ: 結論に至った経緯と議論は tmp/agenticCodingTalking.html 参照（git 管理外）。

1. 役割分担

図1: 主エージェントがハブ。配下サブエージェントは互いに直接やり取りしない。
  • 主エージェントがハブ。サブエージェントは互いに直接やり取りせず、連携と 調停は主エージェントを経由する（各サブエージェントは独立コンテキストで1回走り、結果を返して消える）。
  • 重要なトレードオフ以外は主エージェントが解決する。判断が分岐して成果物が 変わる／設計変更／破壊的変更のときだけ貴殿にエスカレーションする。

2. パイプライン（back-edge を持つ）

図2: 差し戻し辺を持つ自走パイプライン。停止点は3つ。

人間（貴殿）の停止点は3つ:

  1. 設計後レビュー直後 — designReviewer の判定 要相談 時、または重大な未解決指摘あり
  2. 設計差し戻しが起きたとき（実装/テスト中に設計の不備を発見した場合）
  3. 最終レビューの結果

それ以外（実装ループ内の red 修正・軽微な判断）は主エージェントに委譲。

3. レビューの差し込み方 — ハイブリッド

軽い自動ゲートを各境界に置き、最後に多観点レビューを1回。

位置内容担当性質
設計直後5観点の軽量チェック ＋ 最大2往復のラリーdesignReviewer安価・前向き
実装直後テスト緑チェック（門番ではなくループ終了条件）主エージェント自動
テスト直後多観点（役割・テスト十分性・配布/セキュリティ）の並列レビューreviewer（将来）高価・最後に1回

Green チェックの分岐（red になったとき）

red の原因取るべき行動
新規コードの単純バグ修正してループ（回数上限つき）
テストが古い仕様を守っているテストを直す（「甘くして通す」不正は最終 reviewer がフラグ）
失敗が設計の不備を示す設計差し戻し（停止点 2 へ）
環境 / flakychurn せず報告

4. designReviewer のラリー（最大2往復）

図3: designReviewer のラリー手順。記録はすべて designReview.md。
  1. designReviewer が指摘を docs/design/<対象ディレクトリ>/designReview.md に追記（見出し ## レビュー N 巡目、判定 GO / 要相談、 重大/中/軽の3段階、各指摘に - 実装者回答: の空欄を添える）。
  2. 実装者（worker または主エージェント）が、各指摘に 修正 / 反論（理由つき）を同じ md に追記。
  3. designReviewer が再確認 → 解決なら終了、未解決なら2巡目。
  4. 2往復しても未解決の重大指摘が残れば貴殿へエスカレーション（主エージェントが上げる）。

反論が妥当なら指摘は取り下げる（設計原則と実ファイルの根拠で判断、メンツで残さない）。

5. メモリ（教訓）

各エージェントは claudeKnowledge/<agentName>/memory.md を持つ。

  • 起動時: 自分のメモリを読み、過去の教訓を今回の作業に活かす。
  • 終了時: 再現性のある教訓を最後の行動として追記する（サブエージェントには 「閉じる瞬間」を捉えるイベントが無いため、最終行動として書く規律で担保。取りこぼしは主エージェントが backstop）。
  • 抽象度: 残すのは「繰り返す落とし穴・プロジェクト/利用者の好み・有効な勘所」。 コードや git が既に持つ具体（ファイル名・行・個別修正）は残さない。
  • 書込スコープ: 各エージェントが書いてよいのは、自分の memory.md と 役割上の成果物のみ（例: designReviewer は対象の designReview.md）。

詳細規約は claudeKnowledge/README.md を参照。
