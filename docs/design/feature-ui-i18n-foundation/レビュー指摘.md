## [重大1] 【修正】 モジュールレベルで `tr()` を評価しており、言語切替が構造的に効かない

### 事実
- `auth_display.py:17-18`
  ```python
  AUTH_MISSING_TEXT = tr("auth.missing")
  AUTH_UNVERIFIED_TEXT = tr("auth.unverified")
  ```
- `backend_display.py:22,26-29,34`
  ```python
  TTS_NONE_DISPLAY = tr("backend.tts_none")
  CAPTURE_KIND_LABELS = {CaptureKind.DEVICE: tr("capture_kind.device"),
                         CaptureKind.PROCESS: tr("capture_kind.process")}
  SKIPPED_STATUS_TEXT = tr("backend.skipped_status")
  ```
- `status_summary.py` はこの `AUTH_MISSING_TEXT` / `AUTH_UNVERIFIED_TEXT` を import して使用 →
  固定が連鎖する。

### 何が問題か
これらは **import 時に一度だけ** `current_locale()`(=今は ja)で評価され、以後その値に固定される。
Phase 4 で `current_locale()` を実行中に切り替えても、`auth_status_override()` /
`tts_internal_to_display()` 等は **焼かれた定数を返すだけで言語が切り替わらない**。

つまり本ブランチが掲げる「`current_locale()` を将来の多言語の差し込み口にする」という目的を、
土台のうち 2 モジュールが構造的に裏切っている。

さらに **一貫性が無い**: `ready_state` / `language_choices` / `accel_summary` /
`status_summary`(セクション見出し)は関数内で `tr()` を呼ぶため切替に追従する。
同じ logic 層なのに「追従するモジュール」と「焼き付くモジュール」が混在しており、
Phase 4 で「どれが固定でどれが動的か」を再調査する負債になる。

### 推奨
- モジュール定数化をやめ、**表示する瞬間(関数内)で `tr()` を呼ぶ**形に統一する。
  - `AUTH_MISSING_TEXT` 定数 → `auth_status_override()` 内で `tr("auth.missing")` を直接呼ぶ。
  - `CAPTURE_KIND_LABELS` 定数 dict → kind を受け取り `tr()` を返すアクセサ関数に。
- 健全性検査に **「モジュールトップレベルでの `tr()` 呼び出しを禁止」** を追加し、再発を機械的に防ぐ
  (AST で `Module.body` 直下〜代入右辺の `tr()` Call を検出)。

**判断根拠**: 指摘は正当。実コードで `auth_display.py:17-18` と
`backend_display.py:22,26-29,34` がモジュールレベル評価、`ready_state` /
`language_choices` / `accel_summary` / `status_summary` が関数内 `tr()` であることを確認。
同一 logic 層内の不統一は事実で、Phase 4 で `current_locale()` を可変化しても焼き付き定数は
追従しない。本ブランチの存在意義(切替の差し込み口)に直接反するため要修正。
なお `backend_display.TTS_NONE_DISPLAY` は内部値比較(`tts_display_to_internal`)にも使うが、
保存は internal 値で行うため関数内 `tr()` 化しても挙動に影響しない。

## [重大2] 【修正】 `restart_messages.py` がスコープ未達。健全性検査が置換漏れを検出できない

### 事実
- Plan.md:50-51 と testPlan.md:24 は、置換対象に **`restart_messages` を明示的に挙げている**。
- しかし `restart_messages.py` は日本語文言を持ったまま diff に一切現れない(=未着手):
  - `restart_messages.py:12-13` `{"input": "入力", "output": "出力"}`
  - `restart_messages.py:24` `f"{device_label(device_key)}デバイスを切り替えました(再開中…)"`
  - `restart_messages.py:30` `f"...デバイス変更後の再開に失敗しました: {message}"`
- それでも **`pytest` は green(1375 passed)**。

### 何が問題か
「Plan が約束した作業が漏れている」こと自体に加え、**それを検査がまったく捕まえられない**ことが
本質的な欠陥。現状の健全性検査は次の 3 つだけ:
- (a) 欠落キー(コードで使うが辞書に無い)
- (b) 死にキー(辞書にあるが未使用)
- (c) 動的キー(リテラルでない第一引数)

これらは **すでに `tr()` 化された箇所**しか見ない。`tr()` 化されずに残ったハードコード文言
(=置換漏れ)は (a)(b)(c) のどれにも引っかからず、静かにすり抜ける。`restart_messages` は
まさにこの穴に落ちた実例。

Plan は「文言が増えるほど目視突合は破綻するから自動検査を入れる」と述べているのに、
**最も発生しやすい失敗(置換漏れ)を自動検出していない**。これは検査設計の論理的な抜け。

### 推奨
- まず `restart_messages` を tr() 化してスコープを満たす(`restart.device.input` /
  `restart.device.output` / `restart.started` / `restart.failed` 等)。
- 検査を対称化する: **「UI 文言を持つ層(gui/ 配下、i18n カタログ除く)に CJK を含む文字列
  リテラルが直書きされていたら fail」** を追加。未置換の widget は許可リストで段階的に潰す。
  これで Phase 2/3 の置換進捗が定量化でき、漏れは検査が教えてくれる。

**判断根拠**: 指摘は正当。`restart_messages.py` に日本語直書きが残存(L12-13 / L24 / L30)し
diff に現れない=スコープ未達を実確認。健全性検査は tr() 済み箇所のキーしか走査せず、未 tr の
ハードコード残存は欠落/死に/動的のどれにも当たらず原理的に検出不能。`restart_messages` の
tr() 化は必須(Plan の約束を満たす)。加えて CJK 直書き残存検査の追加は、検査を「使われたキー」
だけでなく「置換漏れ」も見る非対称解消として妥当で、Phase 2/3 の進捗計測にもなる。

## [中2] 【修正】 テンプレート引数の静的検査が無い

- `tr(key, **kwargs)` の kwargs 名が、テンプレートの `{placeholder}` と一致するかは **実行時のみ**
  判定される。AST 検査はキーの存在だけ見て、引数名を突合しない。
- 例: `language.src_fallback` は `{old}{new}{backend}{code}` を要求。将来テンプレートか呼び出しの
  どちらかがズレても、その関数を実際に呼ぶ test が無ければ気づけない(重大2 と同種の「実行しないと
  分からない」穴)。
- 推奨: AST で「各 `tr("key", ...)` 呼び出しの kwargs 名 ⊇ 当該テンプレートの placeholder 名」を
  突合する検査を土台に追加。

**判断根拠**: 修正(優先度中)。現状 `format_*` 等のテンプレート関数は logic test
(`test_logic_language_choices.py` 等)で実行経路を持ち、引数不一致は実行時 `KeyError` で
test が落ちるため**今は実害なし**。ただし新規 tr() 追加時に test を書き忘れると重大2 と同型で
すり抜ける。AST の placeholder ⊇ kwargs 突合は安価な恒久防具で、tr() が激増する Phase 2/3 の
前に入れる価値がある。マージ必須ではないが土台に含めると安い。

## [中3] 【修正不要】 カタログ間整合検査が無い

- 「辞書を足すだけで言語が増える構造」を掲げるが、足したとき **ja にあって en に無いキー
  (翻訳漏れ)** を守る検査が無い。今は ja 単一なので潜在化しているだけ。
- 推奨: 「全ロケールのキー集合が一致する」検査の枠を今のうちに入れておく(Phase 4 で効く防具を
  コストの低い今作る)。

**判断根拠**: 修正不要(現フェーズでは)。ja 単一の現状では検査対象が 1 ロケールで自明にパスし、
検出能力がゼロ。`current_locale()` も "ja" 固定で多言語切替の配線自体が無い。en 追加(Phase 4)と
同時に導入するのが自然で、いま「枠だけ」作るのは早すぎる抽象化(CLAUDE.md「動かしてから整理 /
早すぎる抽象化を避ける」)。代わりに Phase 4 のチェックリストへ項目として記録し、忘れない形にする。

## [中4] 【修正】 `auth.missing` / `auth.unverified` は別の真実の源と二重管理になっている

- `messages.py:54-55` の `auth.missing="Missing Credentials"` は `ModelStatus.MISSING_CREDENTIALS.value`
  と**意図的に同一**にしてある(`test_logic_auth_display.py:39-41` がこの一致を検証)。
- つまりこの文言は「インスタンス由来表示(enum value)と揃えるため**翻訳してはいけない**文言」。
  それを i18n カタログに入れると、多言語化時に「ja カタログだけ訳す → enum value とズレる」か
  「訳さない → カタログに英語固定値が居座る」かのジレンマになる。
- 論点: `ModelStatus.value` 側を i18n しない限り、`auth.*` をカタログに置く意味は薄い。
  カタログに入れる/入れないの線引き(=「翻訳対象の文言」か「他の真実の源のミラー」か)を
  土台の段階で決めておくべき。

**判断根拠**: 指摘は正当。`test_logic_auth_display.py:41` が
`AUTH_MISSING_TEXT == ModelStatus.MISSING_CREDENTIALS.value` を固定していることを実確認。
`auth.*` は「翻訳対象」ではなく enum value のミラーで、カタログに置くと二重管理になり、
`ModelStatus.value` 自体を i18n しない限りカタログに置く利得が無い。修正方針: `auth.*` を
カタログから外し `auth_display` 内で `ModelStatus.*.value` を直接参照(単一の真実源に戻す)。
あわせて「カタログに入れる文言 = 翻訳対象のみ。他の真実の源のミラーは入れない」線引きを Plan に明記。

## Plan へのより良い方法(そもそも論)

方式(案1)は変えなくてよい。改善は **「土台の作り込み順序」と「フェーズの切り方」** にある。

> **評価サマリ**: A=採用 / B=採用 / C=採用(計画反映)/ D=部分採用(テンプレ引数検査[中2]は採用、
> カタログ整合検査[中3]は Phase 4 へ見送り)/ E=採用(計画反映)。

### 提案A【採用】: 「切替原則」を土台の規約に格上げする
- 規約「**`tr()` は表示する瞬間に呼ぶ。モジュールレベルで結果を定数に焼かない**」を明文化し、
  健全性検査で機械的に守る(重大1 の恒久対策)。これで土台のまま Phase 4 の切替が効く。

### 提案B【採用】: 検査を対称化し「置換漏れ」を可視化する(重大2 の恒久対策)
- 「ハードコード文言残存検出(gui/ 配下の CJK 文字列リテラル、カタログ除く、許可リスト方式)」を
  土台に含める。Plan の動機「目視突合は破綻する」の論理的帰結。
- 効果: Phase 2/3 の置換を「許可リストを 0 に近づける作業」として定量管理できる。

### 提案C【採用(計画反映)】: Phase 1 を「土台」と「適用」に割る
- 現状 Phase 1 =「土台 + logic 層**全部**」だが、適用が中途半端(restart 漏れ)。
- 案: **Phase 1a = 土台(messages + 4 検査: 欠落 / 死に / 動的 / ハードコード残存)**、
  **Phase 1b = logic 層の置換を残存検査が 0 になるまで機械的に**。
- 「土台を作るブランチ」なら土台(仕組み + 検査)の完成度を最優先にし、適用範囲は検査で保証された
  機械作業に落とすのが、漏れの出ない順序。

## [4a-1]【重大】 Panel 再構築で購読を解除せず、死んだ Panel がリーク + emit のたびに呼ばれる

### 事実
- `main_window.py:114-122` `_rebuild_panels` は各 Panel を `panel.destroy()` するだけ。
- `control_panel.py:73-80` は `self._subscriptions = [...]` で **6 本**(status / text_ready /
  utterance_done / fatal / warn / settings)を購読。`settings_panel.py:151-156` は **4 本**
  (status / restart / settings / running)を購読。
- **どちらの Panel も `destroy()` を override しておらず、`_subscriptions` を `unsubscribe()`
  していない**(grep で確認。`unsubscribe()` を呼ぶのは `layer_settings_dialog.py:587` のダイアログ
  のみ)。
- `app_controller.py:225-235` `_emit` は **登録された全 callback を呼び、各 callback の例外は
  `try/except` でログするだけで他を止めない**。`Subscription`(`backend_base.py:39-64`)は
  `unsubscribe()` を呼ぶまで `_ui_listeners` から消えない。

### 何が問題か
言語を 1 回切り替えるたびに、古い ControlPanel(6 本)+ SettingsPanel(4 本)= **10 本の listener が
`_ui_listeners` に残留**する。新しい Panel もまた 10 本登録するので、N 回切り替えると `(N+1)×10` 本に
膨らむ。帰結は 2 つ:

1. **メモリリーク**: `_ui_listeners[token]` が bound method(`self._on_status_from_thread` 等)を
   保持し続けるため、`destroy()` 済みの Panel と**それが参照する widget ツリー全体が GC されない**。
2. **死んだ widget へのコールバック**: backend が status / running / settings / fatal を `_emit` する
   たびに、破棄済み Panel のハンドラも呼ばれる。ハンドラは `self.after(0, ...)`(破棄済み widget の
   `after`)を叩き **TclError**(`invalid command name`)になる。これは `_emit` の `try/except` で
   握りつぶされ、**app.log にスタックトレースが噴出するだけでクラッシュはしない** → 表向き動くので
   見逃されやすい。

最も痛いのは構図の再来である。`Subscription` の docstring(`backend_base.py:42-43`)は
「**ダイアログの dismiss 等で明示的に `unsubscribe()` してリーク・死んだ widget 参照を防ぐ(R2-6)**」と
明記している。`LayerSettingsDialog` はこれを守って解除しているのに、**毎回 destroy されることが前提の
再構築 Panel 側がこの解除を怠っている**。前回 重大1(土台が「将来の切替の差し込み口」という自分の
存在意義を裏切る)と同型で、今度は「再構築方式が Subscription 機構の前提を裏切る」。

### 推奨
- 各 Panel に `destroy()` override を入れ、`for s in self._subscriptions: s.unsubscribe()` を行ってから
  `super().destroy()` する(ControlPanel は周期 `after`(`_schedule_status_refresh`)も
  `after_cancel` する)。
- 「**購読する Panel は破棄時に必ず解除する**」を規約化し、健全性検査で機械化する
  (例: AST で「`_subscriptions =` を代入するクラスは `destroy` を override し
  `unsubscribe` を呼ぶ」を要求)。前回の「検査で再発を機械担保」の思想と一致。
- そもそも全 Panel 再構築方式が最善かは [4a-6] / Plan 提案で再考する。

### 判断根拠
指摘は正当。`_rebuild_panels` が `destroy()` のみ・両 Panel が `_subscriptions` を解除しない・`_emit`
が例外を握りつぶす、の 3 点を実コードで確認。3 つが揃うと「listener が残る → 死んだ Panel がリーク +
毎 emit で呼ばれる」が必然的に起きる。動作はクラッシュしないが、リークとログ汚染は実害で、切替を
繰り返すドッグフーディングで顕在化する。**マージ前の解消を推奨**。

## Plan へのより良い方法(そもそも論・4a/4b 版)

方式(案1)は変えなくてよい。改善は **「再構築方式の正しさの担保」と「切替の Done 定義」** にある。

### 提案1【強く推奨】: 再構築方式を採るなら「購読の自動解除」を Panel の契約に格上げする
- `_subscriptions` を持つ Panel は `destroy()` で必ず一括 `unsubscribe()`(+ `after_cancel`)する、を
  土台規約にし、AST 検査で機械化する(4a-1 の恒久対策)。前回「検査で再発防止」の延長。

### 提案2【検討】: 「全 Panel 再構築」と「その場 retranslate」を比較してから確定する
- 全 destroy→再生成は重く、購読解除義務・`banner._before_widget` 注入・選択値の ConfigStore 往復復元
  など副作用が多い。代替は **retranslate 方式**(`tr()` で widget を作る時に `(widget, key)` を控え、
  切替時に登録済み widget の text を `tr(key)` で貼り直す。Qt の `retranslateUi` 相当)。リーク無し・
  状態保持・banner 注入不要。ただし動的候補(言語プルダウン等)の貼り直しは別途要る。
- どちらを選ぶにせよ、**当初イベント案からの変更も含めてトレードオフを Plan に 1 表で残す**(4a-6)。

### 提案3【推奨】: Phase 4a の「E2E で通す」に Done 定義(テスト)を与える
- 提案E は「E2E で通す」と言うが合格条件が無い。最小 smoke 3 本([4a-2] 推奨)を Done 条件として
  明記し、`_on_locale_changed` の判断を logic に出して([4a-3])モック不要で検証可能にする。

### 提案4【申し送り】: zh/es 追加(4b 残)の前に翻訳品質の担当・基準を決める
- en は構造検査(parity)は通るが**意味の正しさは未保証**(メモリにも「翻訳品質は未レビュー」)。
  zh/es を機械的に足す前に、誰がどう品質を見るか(ネイティブ確認 / 後回しの明示)を Plan に書いておく。
  本レビューのスコープ外だが、parity が緑でも品質は緑にならない点は土台の限界として記録すべき。
